## 当前实现补记（2026-04-21）

当前代码已落地以下主干契约：

- `/search`、`/entries/{name}` 与首页内搜索点击统一走 `focus graph payload -> select -> Spatial Card` 链路
- `System Overlay` 已承载语言、主题、reduced motion、budget、快捷键与帮助
- `Spatial Card` 改为左右自适应停靠；巡游摘要改为跟随目标停靠
- 提交成功反馈以人类可读消息 + 收件编号呈现，前台不再依赖 `item_id` 文案拼接
- 公开前台语义已收束为五类宇宙分类，不再直接显示内部类型名

当前仍保留的过渡态：

- 渲染主干仍以 2D canvas 为正式 `standard / safe_fallback` 实现
- `WebGPU + Worker + OffscreenCanvas` 仍作为增强型 `immersive` 路线，不作为当前前台可用性的前置条件

## 1. 这次演进要解决什么问题

当前前台虽然已经把 `知识宇宙` 提升为首页 hero 和独立图页，但整体心智仍然是传统门户：

- 首页仍然是“页面 + 模块”
- 搜索、条目、提交、教程仍然分散在不同页面
- 宇宙更像“一个高级浏览器”，而不是“整个系统的操作系统”
- 用户的第一感知仍然是功能入口，而不是“知识正在形成的宇宙”

这会导致三个问题：

1. **世界观断裂**  
   用户先进入传统网站，再跳进宇宙，前后台和前台内部的氛围切换过于生硬。
2. **任务路径割裂**  
   搜索、阅读、提交、学习教程都需要离开当前宇宙语境，空间记忆和沉浸感无法累积。
3. **产品表达偏弱**  
   Sediment 的核心不是“知识网站”，而是“知识沉积、凝聚、成形”。传统门户语义无法充分表达这一点。

因此 v2 的目标不是把图页做得更酷，而是完成一次前台范式切换：

**用户一打开网页，就已经身处知识宇宙之中；门户前台的所有功能，都在这个宇宙里完成。**

## 2. 产品决策

### 2.1 前后台边界

- 前台（portal）整体宇宙化
- 后台（admin）继续保持治理工作台形态
- portal 与 admin 继续共用同一知识数据与图模型，但不共用视觉形态
- 任何后台治理语汇（如 review、job、owner-only、repo lock）都不直接暴露到前台宇宙中

### 2.1.4 知识源与 Agent 边界

- 后端技术栈继续保持当前路线，不为 v2 宇宙前台重写知识存储体系
- `Markdown` 明文知识库继续是核心知识源
- `Obsidian` 可视化与文件工作流继续是项目核心能力，不被前台宇宙替代
- Universe OS 的职责是为 **人类使用者** 提供更自然、更沉浸、更强空间感的知识入口
- Agent 侧继续通过 `MCP / Skill` 接入，不把宇宙前台强行改造成 Agent 主接口
- 若需要更高性能的数据组织，只允许在 Markdown 源之上构建 **派生索引 / 派生缓存**，不允许引入与 Markdown 并列的第二真相源

### 2.1.1 总体美学立场

- 本系统的第一要务是 **科幻浪漫主义**
- 所有交互、镜头、文字、配色、信息排布，都优先服务“浩瀚、诗性、探索欲”而不是“工具效率感”
- 浪漫不等于混乱；宇宙必须在强叙事之上保持稳定、可导航、可定位
- 不采用“企业控制台披一层星空皮肤”的方案；必须让用户感到自己真的进入了一片知识宇宙

### 2.1.2 视觉基调

- 主色气质采用 **深蓝冷黑 + 金色文明感**
- 背景宇宙以冷黑、深蓝、极深青为底，不采用高饱和彩虹星云
- 重要节点、文明感结构、关键路径高亮可使用克制的金色 / 琥珀金
- 不追求霓虹赛博感，而是追求静谧、古老、神秘、带有文明尺度感的宇宙

### 2.1.3 声音原则

- 只使用音效，不使用旁白
- 不使用解释型语音、解说型提示音或高密度 UI 点击噪音
- 音效方向以静谧感、神秘感、空间感为核心
- 开场跃迁、电影化运镜、聚焦、清屏、巡游开始、投放知识种子等关键动作可拥有低频、稀疏、克制的反馈音效
- 默认不依赖强旋律背景音乐来支撑情绪
- 允许存在极轻的、非旋律化环境底噪，用来提供空间感与静谧感
- 默认开启声音
- 右上角提供常驻静音按钮，允许用户随时一键静音
- 如果浏览器拦截自动播放，则使用一个极轻的 `tap to enable sound` 提示，不破坏沉浸感

### 2.2 前台总入口

- `/` 成为唯一的前台主入口
- 用户首屏直接进入知识宇宙，不再先看到传统首页骨架
- 开场首先出现宇宙场景，然后渐显一句诗性引导文案，再淡入 HUD 与功能入口
- 诗性引导文案必须由 locale 模板生成，不能把某一语言硬编码到模板或贴图素材中

### 2.2.3 终端策略

- v2 首发 **不支持移动端**
- 移动端或窄屏访问时，不尝试提供功能残缺的宇宙交互
- 直接显示明确提示：请使用电脑端访问
- iPad 横屏与触屏 Windows 设备视作桌面端能力范围
- 因此前台仍需要考虑鼠标 + 触摸的双输入逻辑，但不需要考虑手机式窄屏布局

引导文案契约采用模板化表达，例如：

`请探索这片知识宇宙，{knowledge_base_name}`

或

`Explore this knowledge universe, {knowledge_base_name}`

文案模板本身属于 UI locale 数据，不属于视觉素材。

### 2.2.0 诗性文案池

- 开场诗性文案来自一个约 `100` 条的候选池
- 每次访问随机抽取一条
- 文案只承担情绪建立，不承担功能说明
- 开场只显示一句诗性句子，不再叠加第二句功能说明

### 2.2.1 首次访问开场镜头

首次访问必须播放一个 **2 到 3 秒、不可打断、完成度极高** 的开场镜头。

推荐镜头脚本：

1. 黑场或近黑场开始，只保留极弱星尘与远处噪声
2. 镜头以“宇宙飞船跃迁”感冲出一段扭曲星流，进入一片新的星海
3. 画面稳定后，右下方出现一颗占画面约 `1/4` 的稳定知识星球，缓慢自转
4. 远处存在稀疏而深邃的知识星群，强调“宇宙很大，当前只看到极小一角”
5. 屏幕中央或偏上方缓缓浮现诗性引导文案
6. 文案停留极短时间后淡出，HUD 与可操作能力再延迟淡入

开场停留星球的选择规则：

- 只从稳定知识星球中选择
- 采用“代表性 + 可读性 + 视觉完成度”的加权随机
- 不采用完全随机，以避免落到难以理解或画面效果不佳的目标

开场停留星球的信息呈现：

- 在该 `1/4` 星球附近显示它对应的真实知识标题
- 同时显示一句真实知识摘要
- 这段内容不应喧宾夺主，只用于让用户立即感到“这里装的是真知识，不是抽象概念秀”

设计约束：

- 首次开场不可打断
- 回访时允许跳过，默认直接进入可操作态
- 开场镜头不是 logo 动画，而是“你刚抵达一片知识星海”的到达感
- 开场镜头中不得出现大量 UI 杂讯、调试信息或菜单元素

### 2.2.2 回访策略

- 通过本地状态判断是否为首次访问
- 首次访问播放完整开场镜头
- 回访默认跳过，直接进入 `roaming`
- 设置中允许用户手动“再次播放开场”

### 2.2.4 前 30 秒主叙事

前台首个 `30 秒` 的体验主线明确采用：

- **1：史诗抵达型**
- **3：任务导航型**

因此首个半分钟必须连续完成三件事：

1. 让用户相信自己刚刚抵达一片真正的知识宇宙
2. 让用户看见一个具体、可理解、可被命名的知识对象
3. 让用户迅速意识到：自己不仅能沉浸，还能找到目标并继续探索

这意味着 v2 首屏不应只追求“美”，还必须把“可导航”自然埋进抵达体验里。

具体节奏：

- 前 `2 到 3 秒`：完成史诗抵达，建立世界观与尺度感
- 接下来的数秒：让用户看见一个真实知识星球与其具体知识内容线索
- 再之后：通过 HUD、`survey` 或焦点提示，让用户理解“我现在可以寻找目标、探索关联”

### 2.3 路由语义

前台仍保留历史 URL，但不再维护独立页面壳，而是全部映射到同一个 `Universe OS` 页面壳：

| 路由 | v2 行为 |
| --- | --- |
| `/` | 打开宇宙默认探索态 |
| `/search?q=` | 打开宇宙 `survey` 搜索态，并把结果锚定到俯瞰导航视角 |
| `/entries/{name}` | 打开宇宙并直接飞向对应节点，展开详情弹层 |
| `/submit` | 打开宇宙并直接展开提交面板 |
| `/tutorial` | 打开宇宙并直接展开教程 / onboarding 面板 |
| `/portal/graph-view` | 打开同一 `Universe OS`，但以更纯粹的沉浸漫游预设启动：默认跳过开场、HUD 更弱、保留次级 `Open Quartz` 入口 |

结论：

- 前台仍支持深链
- 但前台不再依赖“切换到另一张页面”完成主任务
- URL 是状态入口，不是视觉壳的分割器

### 2.3.1 与既有设计的冲突收束

v2 与当前 `current` 设计并不是“局部增强”关系，而是一次 **前台契约重写**。为了避免旧文档与新蓝图互相打架，这里明确收束：

| 冲突面 | 当前设计 | v2 收束方案 |
| --- | --- | --- |
| 前台页面形态 | `/search`、`/tutorial`、`/entries/{name}`、`/submit` 都是独立页面 | 这些路由继续存在，但都改为同一 `Universe OS` 壳上的深链状态 |
| `/portal/graph-view`` 语义 | 独立沉浸图页 | 保留 URL，但改为同一壳的“沉浸漫游预设”，不再维护第二套前台结构 |
| portal / admin 共享 shell | 当前设计要求共享基础 shell | v2 只保留共享 theme、locale、能力和少数基础组件；portal 不再与 admin 共享结构壳 |
| 节点点击交互 | 当前图页强调右侧 `focus sheet` | v2 由 `Spatial Card` 统一承接，并提供 `compact focus -> expanded reader` 两级 |
| Quartz 在前台的位置 | 当前前台主导航保留 Quartz 直达 | v2 保留 `/quartz/` 与直接访问能力，但降级为次级入口，不再与宇宙主入口等权竞争 |

由此产生的文档处理原则：

- `design/current/web-surfaces.md` 中关于前台独立页面骨架、可见 page-level heading、共享基础 shell 的前台部分，会被 v2 前台契约显式取代
- `design/current/platform-architecture.md` 中关于前台 IA 的路由列表仍可保留，但其页面形态解释需要改写为“单一宇宙壳 + 深链状态”
- Quartz 托管、Quartz CSP、Quartz 图谱兼容性与后台 Quartz 管理契约继续有效，不被 v2 推翻
- Agent 相关的 MCP / Skill 教程契约继续有效，但承载位置从独立教程页转为宇宙内教程面板

对测试的影响也必须显式承认：

- 现有锁定首页、搜索页、教程页、提交页和条目页独立 HTML 结构的测试，若照 v2 落地，将不再代表新前台契约
- 这些测试不是“小修补”，而是需要按 `Universe OS` 新契约整体重写

## 3. Universe OS 的页面壳

### 3.1 固定骨架

前台统一采用一个沉浸式宇宙壳，包含以下稳定区域：

1. **Cosmos Stage**
   - 全屏或近全屏 3D 场景
   - 承担星图、飞行、聚焦、巡游、路径高亮、景深和 dimming
2. **Intro Layer**
   - 开场引导文案
   - 轻量品牌标识
   - 首次进入时渐显，用户交互后淡出或收敛
3. **HUD Layer**
   - 搜索 / 俯瞰星图、热点、探索、巡游、过滤、budget、后退、状态行、帮助
4. **Overlay Layer**
   - 节点详情、搜索结果、提交、教程、系统设置等浮层
5. **Navigation Aids**
   - 全屏俯瞰星图态
   - 焦点路径提示
   - 当前模式与状态提示

补充约束：

- `Universe OS` 必须拥有独立的前台表面语言
- 它可以继续复用 theme、locale、上传能力与少量基础 utility
- 但不得继承旧 portal / admin 的 `card / panel / button / page shell` 结构语义来塑造主要视觉体验
- 换言之，前台宇宙不是“旧门户换皮”，而是一套独立的交互外壳

### 3.1.1 HUD 默认策略

- HUD 默认采用极简常驻模式
- 只暴露最少量的高频入口与状态提示
- 用户可手动展开 HUD，以获取完整功能
- 这种展开不是切换页面，而是同一宇宙壳中的信息层级提升

### 3.2 页面状态机

统一页面状态机：

`intro -> roaming -> focused -> overlay_open -> roaming`

补充状态：

- `searching`
- `flying_short`
- `flying_cinematic`
- `survey`
- `cruising`
- `submitting`
- `fallback_2d`

约束：

- 打开 overlay 不得重建场景
- 切换路由 query 不得重建节点坐标
- 搜索、详情、提交、教程都只是 `Universe OS` 的状态切换

### 3.3 清屏态

前台必须支持显式 `清屏`：

- 隐藏绝大部分 HUD
- 隐藏 overlay
- 保留极少量退出提示或状态提示
- 让用户获得更强的沉浸感和更完整的宇宙画面

清屏态不是退出产品，而是进入更纯粹的宇宙观看模式。

## 4. 宇宙内功能如何吸收现有前台能力

### 4.1 搜索

搜索不再是传统页面中的表单，而是宇宙中的“导航指令”。

契约：

- 点击搜索框时，镜头先主动拉远，进入全屏 `survey` 俯瞰星图态
- 搜索并不是在当前画面上叠一个局部弹窗，而是让当前宇宙场景切换到俯瞰导航视角
- 搜索与俯瞰星图共用同一入口和同一状态机
- 搜索能力覆盖：
  - 标题匹配
  - alias 匹配
  - 全文结果召回
- 不把自然语言问答并入搜索框主语义
- 输入时显示建议候选
- 选择候选时，优先判断目标是否已在当前视野数据中
- 已在图内：直接飞行聚焦
- 不在图内：拉取邻域 / 聚焦 payload，再飞行
- 搜索结果默认在宇宙内以弹层或 palette 形式展示，不离开当前场景
- 搜索匹配到的星球在俯瞰星图内亮起；未匹配节点明显弱化

### 4.1.1 `survey` 俯瞰星图态

`survey` 的相机风格优先采用 **战略星图** 视角：

- 整体更接近高空战略地图，而不是电影镜头的倾斜俯拍
- 优先强调结构可读性、空间定位和目标选择效率
- 允许保留极轻的透视层次，但不让透视变形破坏“读图感”
- 用户进入 `survey` 后应立即理解自己在看一张可导航的宇宙总图

### 4.1.2 `survey` 交互边界

`survey` 必须保持强地图感，因此交互边界固定为：

- 只允许平移
- 只允许缩放
- 只允许点选目标
- 不允许自由旋转场景

补充约束：

- 进入 `survey` 时，搜索框必须显示出来并进入强调态
- 用户应明确知道自己此时处于“俯瞰导航”而不是“沉浸漫游”
- 退出 `survey` 后再回到普通宇宙漫游镜头

搜索结果面板需同时支持两种模式：

- **候选模式**：快速跳到一个已知节点
- **结果模式**：展示多个结果卡片，并允许用户逐个飞去

搜索结果列表布局：

- 结果列表默认停靠在左上角搜索框下方
- 它应与搜索框形成一体化视觉结构
- 结果列表不得占用过多屏幕面积，以免破坏战略星图的总览感

`survey` 节点表达密度：

- 采用中等密度
- 既要保留结构骨架，也要保证足够多的目标可被选中
- 不做“只剩线路图”的极简模式，也不做“全图塞满节点”的密集模式

目标选中后的退出规则：

- 在 `survey` 中选择目标后，立即退出 `survey`
- 直接进入聚焦或电影化运镜
- 不增加额外确认步骤

选中目标后的通用特效：

- 触发一次华丽运镜，把目标节点带到主视觉中心
- 与其直接相连的节点同步亮起
- 其他节点与连线几乎隐去，只保留微弱背景星尘
- 这个“选中焦点特效”不只用于搜索，所有选中路径都共用同一视觉语言

### 4.1.3 选中后的镜头构图

点击或搜索选中某个星球后，镜头构图需要兼顾“主焦点”和“可继续探索的一跳邻域”：

- 目标星球必须成为主视觉中心
- 镜头应尽量同时容纳全部或尽可能多的一跳相连星球
- 二跳和三跳不要求全部纳入首屏，但可以通过路径衰减继续提示其存在
- 如果一跳邻域过大，优先容纳路径重要度更高、视觉权重更高的相连节点

### 4.2 条目详情

独立条目页不再是主入口；条目阅读改为“抵达星球轨道后展开内容”。

主阅读界面为 `Spatial Card`：

- 只在用户明确点击节点后出现
- 固定在稳定阅读区域内，不随星球屏幕投影实时漂移
- 视觉上与目标星球建立明确归属关系
- 默认按目标位置在左右两侧智能切换，优先避免遮挡主焦点与关键邻域
- 显示知识的完整 Markdown 内容
- 阅读窗采用中等尺寸文档窗形态
- 窗口不要过大，避免压住主要宇宙画面
- 内容区允许滚动
- 可收缩
- 至少包含：
  - 标题
  - 节点类型 / 状态
  - 完整正文
  - 邻近知识入口
  - 其余辅助信息

辅助信息策略：

- 所有信息都允许在 `Spatial Card` 中查看
- 不重要或次级信息采用抽屉 / 折叠区收起
- 默认展开的信息只保留最需要读的内容，避免把阅读窗变成后台工作台

规则：

- 单击节点默认打开 `Spatial Card`
- 关闭详情后镜头位置保持不变
- `Spatial Card` 必须可以最小化，避免遮挡宇宙画面
- 用户可通过清屏进一步获得更强沉浸感

相关知识跳转规则：

- 从 `Spatial Card` 点击相关知识时，默认复用当前阅读窗
- 如果目标距离近，则直接平移 / 短距离对焦
- 如果目标距离远，则进入电影化运镜

### 4.3 提交

提交能力改为宇宙内 `Seed Dock`：

- 文本提交
- 文档上传
- 当前选择状态
- 提交成功与失败反馈

叙事语义：

- 用户不是“填写一张表单”
- 而是在向宇宙中投放一颗新的知识种子

约束：

- 失败时必须保留用户输入
- 上传状态与错误必须在当前 overlay 内可见
- 用户提交的内容在通过审核或进入前台可见层之前，不得伪装成已经存在于宇宙中的新星球
- 提交成功后允许用户选择：
  - 回到原位置
  - 飞向“刚刚影响到的知识区域”

### 4.3.1 提交成功反馈

提交成功后，允许出现一个 **短暂出现又消散的种子光点**：

- 它只表达“系统已收到这颗种子”
- 不表达“它已经成为正式星球”
- 光点必须短暂、克制、易于理解
- 消散后不在前台宇宙中留下误导性的常驻实体

### 4.4 教程 / Onboarding

教程页改为宇宙内 `Academy Panel`：

- MCP 接入说明
- Skill 下载入口
- 工具分工说明
- 面向 Agent 的自然语言使用方式

定位：

- 教程不再是一张普通内容页
- 而是“这片宇宙的使用说明书”
- 它是给人类用户使用的常驻可打开面板
- Agent 侧继续通过 Skill / MCP 访问，不把前台教程面板混同为 Agent 主入口

### 4.5 帮助与设置

帮助、语言、性能设置、快捷键总览收束到 `System Overlay`：

- 语言切换
- motion / reduced motion
- budget 档位
- 帮助说明
- 键盘映射

### 4.5.1 帮助面板优先级

帮助面板内容按以下主次展开：

1. 操作方式
2. 搜索 / `survey` / 巡游说明
3. 天体类别含义
4. 提交与教程入口
5. 设置项

补充要求：

- Skill / MCP 的介入教程必须继续保留，不能因为前台宇宙化而丢失旧能力
- 但这部分属于帮助 / 教程中的次级内容，不应压过普通用户的探索路径

### 4.6 标签显示策略

前台默认只显示少量高优先级标签，但不同状态下显示重点不同。

基础规则：

- 常态漫游时，只显示高优先级少量标签
- 屏幕中间 `50%` 区域内的星球标签应尽量优先显示，帮助用户形成“主动移动即可发现更多目标”的探索感
- 屏幕边缘区域标签更克制，避免变成信息墙

状态增强：

- 选中某个知识星球后，该星球标签必须显示
- 被点亮的一跳相关星球标签也应显示
- 二跳 / 三跳标签按路径重要度与当前屏幕空间选择性显示
- `survey` 俯瞰态可显示更多结构性标签，但仍要避免全图铺满文字

总体原则：

- 标签服务探索，不服务罗列
- 宁可少而准，也不要全量铺开

## 5. 手动探索与自动探索

### 5.1 手动探索

宇宙必须支持完整的人工漫游：

- 鼠标拖拽旋转
- 中键平移
- 滚轮缩放
- 键盘探索映射
- 全屏俯瞰星图态中的定位与跳转

原则：

- 视角控制的目标是“探索”
- 不是“摆拍”
- 平移和缩放不能被后台自动纠偏偷偷抵消

### 5.1.1 桌面触摸手势

对于 iPad 横屏与触屏 Windows 设备，采用以下默认手势：

- 漫游态：
  - 单指拖动：旋转 / 轨道漫游
  - 双指拖动：平移
  - 双指捏合：缩放
  - 单击：选中节点
- `survey` 俯瞰态：
  - 单指拖动：平移
  - 双指捏合：缩放
  - 单击：选中节点

约束：

- `survey` 仍然不开放旋转
- 触摸逻辑必须和鼠标逻辑在语义上保持一致，而不是形成两套完全不同的导航方式

### 5.2 自动探索

自动探索不是屏保，而是前台一级能力。

包含三种自动模式：

| 模式 | 触发方式 | 目的 |
| --- | --- | --- |
| 热点跳转 | 用户点击热点 | 快速发现当前最活跃或最未成形的知识 |
| 策略探索 | 用户点击探索 | 沿当前策略挑选下一个值得看的节点 |
| 巡游模式 | 用户清屏或主动开启 | 让系统持续讲述知识宇宙的结构与变化 |

### 5.2.1 巡游路线生成

巡游路线不应完全随机，也不应完全脚本化。

推荐原则：

- 稳定知识与形成中知识交替出现
- 在可解释的范围内保留一定随机性，避免每次巡游都像重复播放同一段录像
- 随机性受知识重要度、近期活跃度、视觉多样性和当前已访问历史共同约束
- 同一轮巡游中避免连续访问过于相似的节点或完全同簇节点
- 可成为巡游停靠点的对象包括：
  - 稳定知识
  - 形成中知识
  - Knowledge Basins
  - Navigation Structures

### 5.3 清屏与巡游模式

“清屏”是 v2 的核心戏剧性动作。

行为：

- HUD 收起到极简态
- overlay 全部关闭
- 场景进入自动巡游
- 镜头缓慢经过热点、盆地、凝聚中的节点和关键知识通路
- 飞掠各个星球时展示其概要
- 状态文案以低干扰方式出现，不得像 PPT 自动播放

中断规则：

- 任意鼠标、键盘或 HUD 交互，立即退出巡游
- 退出后恢复到用户最近的可交互状态

### 5.4 空闲触发巡游

- 当页面超过 `20 秒` 没有用户操作时，自动进入巡游模式
- 无操作的判定至少包括：
  - 鼠标未移动
  - 未点击
  - 未滚轮
  - 未键盘操作
- 自动巡游开始前，可允许一个极短、低干扰的状态提示
- 自动巡游与主动清屏进入的巡游在视觉语言上一致

### 5.5 巡游的主次目标

巡游模式的目标优先级固定为：

1. **A：屏保式沉浸展示**
2. **C：自动发现系统，帮助用户选目标**
3. **B：带旁白感的知识导览**

因此巡游首先要好看、其次要帮助发现、最后才是解释。

### 5.5.1 巡游控制

巡游期间保留极简控制：

- `Pause`
- `Next`
- `Exit`

规则：

- 控件必须极简，不破坏沉浸感
- `Exit` 后镜头停留在当前所处位置，而不是跳回初始位置

### 5.6 巡游概要结构

飞掠各个星球时展示的概要采用双层结构：

1. 诗性一句话
2. 一句事实摘要

要求：

- 诗性句负责氛围和记忆点
- 事实句负责最低限度的信息确认
- 两句都必须短，不得把巡游做成长篇阅读器
- 诗性句来自模板化文案
- 事实句来自真实知识摘要

### 5.6.1 巡游停留节奏

- 巡游停留时长采用动态变化
- 默认节奏约为 `3 秒`
- 重要节点可适度延长，次级节点可适度缩短

### 5.7 巡游概要位置

- 双层概要应靠近当前目标星球出现
- 但不能直接贴附在目标星球表面或遮挡主焦点
- 它更像一个与当前目标保持视觉关联的悬浮信息层

## 6. 运镜系统

### 6.1 运镜分档

运镜必须区分目标距离：

| 档位 | 条件 | 行为 |
| --- | --- | --- |
| 近距对焦 | 目标在当前局部附近 | 快速插值，弱特效 |
| 中距牵引 | 目标在同一知识团但不在当前视野中心 | 短路径高亮，镜头柔和转场 |
| 远距电影化 | 目标远离当前焦点或跨簇 | 触发华丽飞行、路径点亮、无关区域压暗 |

### 6.2 远距电影化运镜

远距飞行的契约：

- 先建立目标路径或引力方向感
- 再推进镜头
- 最后把目标节点拉到主焦点区
- 连接节点同步高亮
- 其他区域进入低亮度 dim 态
- 屏幕上下出现黑边，模拟 `21:9` 电影画幅
- 黑边只在电影化阶段出现，结束后柔和退场

禁止事项：

- 为了炫技绕大弯
- 飞行时间长到打断操作
- 近距离操作也强制套用电影化飞行

### 6.2.2 路径高亮衰减

当焦点节点被选中后，路径高亮按重要度逐层衰减：

- 最多显示 `3` 跳
- 一跳最亮，承担主探索线索
- 二跳明显弱于一跳，但仍可读
- 三跳只保留极弱提示，提醒用户“更远处仍有结构”
- 超过三跳不进入可见高亮层

### 6.2.1 电影化巡游镜头

巡游模式默认采用电影化镜头语言：

- 镜头像宇宙飞船一样飞掠各个知识星球
- 转场优先体现尺度感、留白感和速度层次
- 靠近目标时短暂停留，让用户看见星球形态与概要
- 离开目标时再进入下一段长镜头或跃迁
- 不采用频繁切镜的 MV 式节奏

### 6.3 Reduced Motion

必须支持 reduced motion：

- 关闭电影化远距飞行
- 关闭高幅度景深变化和大面积脉冲
- 保留清晰的状态反馈与路径高亮

## 7. 天体美学与随机生成

### 7.1 设计原则

节点不能只是发光点；每颗“知识星球”都要可辨认、可分类、可生成。

要求：

- 同类型知识必须有稳定的视觉骨架
- 同类型不同节点必须存在 seed 驱动的差异
- 随机性来自确定性 seed，而不是每次刷新随机变脸

### 7.2 类型骨架

建议骨架：

| 节点类型 | 天体表达 |
| --- | --- |
| `canonical_entry` | 稳定行星 / 固态天体 |
| `insight_proposal` | 星云 / 胚胎恒星 / 脉冲体 |
| `query_cluster` | 引力团 / 漩涡 / 磁场团 |
| `cluster_anchor` | 主恒星 / basin |
| `index_segment` | 航标 / 星门 / 轨道结构 |

补充原则：

- 从系统实现视角，所有知识类型都应有对应的天体表达
- 从用户视角，不必把所有内部类型都暴露成独立概念类别
- 可以把若干实现类型收束为更少的前台感知类别，避免增加用户心智负担
- 前台用户感知类别固定收束为 `5` 类：
  - `Stable Knowledge`
  - `Forming Knowledge`
  - `Question Constellations`
  - `Knowledge Basins`
  - `Navigation Structures`
- 系统内部类型名默认不直接暴露给用户，只在帮助与文档说明中解释

### 7.3 生成方式

- 采用“类型骨架 + 参数扰动 + seed”生成 SVG
- SVG 转纹理或图集后进入渲染层
- 在远景只显示轻量 sprite / atlas
- 在近景才切换为更复杂的材质与 halo

约束：

- 视觉素材中不能嵌入语言文本
- 节点外形必须不依赖当前 locale
- locale 差异只体现在 HUD / 文案 / overlay

## 8. 渲染与性能策略

### 8.1 混合分层

首发明确采用“混合分层”策略，而不是纯电影优先：

- 远景：点云 / sprite / atlas
- 中景：轻量 halo、简化连线、低频 pulse
- 近景：高质量天体纹理、局部细节、可读 label

### 8.1.1 双档渲染级别

v2 在用户端默认假设：

- 使用桌面端较新的 Chrome
- 基线目标浏览器版本为 `Chrome 130+`
- 浏览器与设备能够提供 `WebGPU + Worker + OffscreenCanvas` 所需的主干能力

在这个前提下，前台仍然保留两档清晰可理解的渲染级别，但它们表达的是 **支持范围内的效果分层**，而不是“向更差兼容性妥协”的降格模式。

前台面向主流核显、Apple M 系列基础机器与更强设备，默认提供两档清晰可理解的渲染级别：

| 档位 | 目标设备 | 策略 |
| --- | --- | --- |
| `low` | 支持 WebGPU 的核显、轻薄本与基础款机器 | 保留完整宇宙语义，但压低后效、粒子、体积感与局部精细度 |
| `high` | Apple M 系列基础款与更强设备 | 保留完整混合分层、电影化运镜、精致材质与少量高级后效 |

用户能手动切换，系统也可自动降档。

### 8.1.2 渐进式加载与同屏密度

- v2 采用渐进式加载，而不是一次性把所有星球都以高精度推到前景
- 宇宙整体允许很大，但默认视图必须保持稀疏、留白和可呼吸感
- `low` / `high` 两档的目标不是改变同屏节点数量，而是改变渲染精度与特效强度
- 同屏稳定可感知的星球数量目标为 `20 到 30` 颗
- 俯瞰星图态必须保持流畅；在该状态下节点可更多，但主要以轻量表达呈现
- 当用户聚焦某个局部时，再按邻域、重要度和距离逐步加载更精细的表现层

### 8.1.3 分辨率基线

- 系统最低体验基线可下探到 `720p`
- 在 `720p` 下允许视觉效果、留白和信息密度略有退化
- `1080p` 是首选的最佳视觉效果目标分辨率
- 文档中的布局、HUD 密度、阅读窗尺度与电影化构图，默认都以 `1080p` 为主要打磨对象
- 当分辨率下探到 `720p` 时，优先收紧 HUD 与阅读窗尺寸
- 同时同步下调特效强度、动效幅度和非必要视觉装饰
- 不优先减少节点规模，而是先压缩 UI 占用与特效负担

### 8.2 固定坐标

- portal 前台继续使用后端稳定坐标
- 搜索、探索、巡游、深链、回退都只改变镜头与 overlay 状态
- 节点位置不能因交互而漂移

### 8.3 LOD 与预算

保留三档 budget，但 v2 对它们的定义更明确：

| budget | 目标 |
| --- | --- |
| `conservative` | 支持环境中的保守稳定档 |
| `medium` | 默认档；兼顾精致与流畅 |
| `aggressive` | 高性能环境实验档，用于放开更高成本的局部效果 |

自动降级：

- 持续掉帧时自动从 `medium -> conservative`
- 在支持环境内优先通过降低后效、粒子、阴影与标签密度回收预算
- 不再把“不支持现代图形能力”的环境视作 v2 的主要兼容目标
- 降级后仍然保持同一宇宙 UI 语义，不能退回旧门户页面

### 8.3.1 浏览器与能力门槛

v2 的取舍明确偏向 **高性能优先于广覆盖兼容性**：

- 前台正式支持环境以 `Chrome 130+` 桌面端为基线
- 主干能力假定包含：`WebGPU`、`Dedicated Worker`、`OffscreenCanvas`
- 若环境不满足主干能力，不提供“低质量但勉强可用”的宇宙降格体验
- 不满足条件时，直接展示明确的“不受支持环境”说明，而不是回退到旧门户或残缺 3D
- 面向用户的原则是：**要么提供我们认可的宇宙体验，要么明确说明当前环境不受支持**

### 8.4 2D Fallback

在当前更激进的技术取舍下，`2D fallback` 不再是用户主路径，而只保留为：

- 内部调试模式
- 自动化测试辅助视图
- 极端故障下的开发诊断工具

结论：

- 用户侧不再以 `2D universe lite` 作为正式兜底体验
- 若 `WebGPU + Worker + OffscreenCanvas` 主干不可用，直接进入不支持提示页
- `2D fallback` 的存在意义是帮助研发与测试定位问题，而不是扩展正式支持矩阵

### 8.5 技术路线判断

用户倾向保留当前 “Python 页面壳 + 前端运行时” 模式，只要它的性能能与完整前端重构方案媲美。

v2 的技术判断是：

- **A 本身不是性能瓶颈**
- 真正的性能瓶颈来自：
  - 图渲染内核
  - 数据量与 LOD
  - 材质 / shader / 后处理
  - 贴图与 SVG 资产管线
- 因此前台即使不改为大型 SPA，也可以在渲染性能上达到甚至超过框架化方案

结论：

- v2 优先采用 **A-prime 路线**
- 即：保留 Python/Starlette 提供页面壳与 bootstrap 数据
- 但前台宇宙运行时重构为自管的高性能场景内核
- 不为了“现代化”而引入重量级前端框架
- 在“用户使用较新的 Chrome 浏览器”假设下，v2 直接采用 **更激进的现代图形主干**
- 图形主干从“`WebGL2` 优先”提升为“`WebGPU` 优先”
- 运行时线程模型从“先主线程、后优化”提升为“`Worker-first`”
- v2 明确接受更窄的正式支持矩阵，以换取更强的峰值表现与更稳的帧预算

### 8.6 推荐技术栈

在当前信息下，推荐技术栈如下：

| 层 | 推荐方案 |
| --- | --- |
| Web 服务层 | Python + Starlette + Uvicorn |
| 页面壳 | 服务端 HTML 模板 + bootstrap JSON |
| 前台运行时 | 原生 ES Modules，新宇宙运行时模块统一使用 TypeScript |
| 3D 渲染 | `three/webgpu` 起步的 WebGPU-first 自管场景 |
| 渲染线程 | `Dedicated Worker + OffscreenCanvas` 主渲染线程模型 |
| 图布局来源 | 后端稳定坐标，不在前端做力学布局 |
| 构建 | `esbuild` 或等价快速打包器 |
| 音效 | 预渲染素材 + Web Audio 控制的混合方案 |
| 输入层 | 鼠标、键盘、桌面触摸统一手势抽象 |
| 文字标签 | `MSDF / SDF atlas` 方案，不依赖 DOM 标签墙或临时 sprite text |
| 贴图生成 | seed 驱动的程序化生成 + atlas / texture cache |
| 数据加载 | bootstrap 骨架 + 邻域 / 视口渐进式加载 |
| 性能监测 | FPS + Long Animation Frames + 自定义镜头 / 加载埋点 |

### 8.6.1 图形能力基线

- 默认图形基线采用 `WebGPU`
- 正式支持环境默认按 `Chrome 130+` 设计
- `WebGL2` 不再作为用户侧正式兜底路径
- 若主干能力不足，直接进入“不受支持环境”提示
- 如果后续需要保留 `WebGL2`，其角色也应是内部调试或过渡阶段的次级实现，而不是用户承诺

### 8.6.2 Worker-first 场景运行时

v2 前台默认采用 `Worker-first` 架构：

- 主线程负责：
  - 服务端页面壳接入
  - HUD / Overlay / Markdown 阅读窗
  - URL、历史记录、locale、可访问性、声音启用手势
- 渲染 Worker 负责：
  - WebGPU 设备与渲染循环
  - 相机插值、巡游运镜、可见性裁剪、LOD 切换
  - 选中态 dim/highlight 分组
  - `survey` 与 `roaming` 共用的场景内核
- 主线程与 Worker 之间通过明确的命令 / 状态通道同步，不允许 UI 层直接越过协议操控场景内部细节

### 8.6.3 高配渲染档位的边界

`high` 档允许引入少量更昂贵但仍受控的视觉能力：

- 更精致的航迹与路径余辉
- 更细腻的 halo / glow 分层
- 克制的体积感与空间雾
- 电影化镜头切换中的轻量后效

但约束仍然成立：

- 不追求大而全的电影后处理堆叠
- 不把 `high` 档做成“华丽但不稳”的 demo 模式
- 一切高级效果都必须让位于镜头稳定性、聚焦清晰度与巡游流畅性

### 8.6.4 性能优先级原则

v2 的技术优先级明确固定为：

`性能 > 质量 > 兼容`

展开为工程约束即：

- 若三者冲突，第一优先保住帧时间与交互稳定性
- 兼容性是最先被牺牲的维度，因此正式支持矩阵被主动收紧
- 质量允许在 `low / high / aggressive` 档位之间分层，而不是要求所有设备都维持同一视觉豪华度
- 不允许为了“看起来更高级”而引入会破坏持续稳定帧率的效果
- 不允许为了“更多机器也能打开”而退回低质量兼容路径，稀释产品心智

### 8.6.5 渲染抽象层

从性能与可控性的综合角度，渲染抽象层采用：

- **`three/webgpu` 起步**
- **为热点模块保留原生 WebGPU escape hatch**

原因：

- 纯 `three/webgpu` 足够快，但在热点场景下自由度仍有限
- 直接全量自写原生 WebGPU 虽然理论上更激进，但会把大量成本消耗在基础设施重写，而不是镜头、巡游、聚焦和宇宙表达本身
- 因此前台应以 `three/webgpu` 负责场景骨架、相机、基础材质与资源生命周期
- 对性能热点保留原生 WebGPU 旁路，例如：
  - 高密度 picking
  - 特殊粒子场
  - 标签批量渲染
  - 定制轨迹或局部特效 pass

### 8.6.6 Shader 与材质路线

材质与 shader 策略采用混合方案：

- 常规业务材质优先使用 `TSL / node material`
- 性能热点与关键视觉特效允许手写 `WGSL`

理由：

- 这样既能保留 three 生态在资源管理与表达层上的便利性
- 又不会把高成本热点锁死在通用材质抽象里
- v2 里真正值得手写 `WGSL` 的对象，预计只包括少数热点：
  - 标签 / glyph 批渲
  - 大规模星尘 / halo / trail
  - 高密度 picking pass
  - 少量电影化后效

### 8.6.7 Worker 拆分粒度

为优先保住性能，运行时采用双 Worker 主结构：

- `render worker`
- `data worker`

职责建议：

- `render worker`
  - WebGPU 初始化与设备生命周期
  - 场景更新、相机插值、可见性裁剪、LOD 切换
  - 渲染命令生成与逐帧绘制
- `data worker`
  - 图块 / 邻域数据解包
  - 搜索结果映射
  - 索引与空间数据结构构建
  - 运行时 atlas / 资源任务调度

不建议首版就继续切成更多 Worker：

- 过多 Worker 会显著增加同步成本与调试复杂度
- 对 v2 这种强交互、强相机驱动场景，双 Worker 已经是性能与复杂度的较优平衡点

### 8.6.8 主线程与 Worker 通信模型

通信模型采用：

- **明确的 command bus**
- **周期性 snapshot state**

即：

- 主线程向 Worker 发送高层命令，而不是直接操控场景细节
- Worker 定期回传精炼后的可消费状态快照，而不是持续抛洒低层事件

推荐命令类别：

- `camera.focus`
- `camera.enter_survey`
- `camera.exit_survey`
- `selection.set`
- `overlay.open`
- `overlay.close`
- `cruise.start`
- `cruise.pause`
- `cruise.next`
- `cruise.exit`
- `perf.set_budget`

推荐快照类别：

- 当前焦点
- 当前模式
- 当前预算档位
- 当前可视邻域摘要
- 当前性能状态
- 当前不支持/降级原因

### 8.6.9 Picking 路线

拾取策略采用混合路径：

- 近景 / 低密度：CPU 空间索引 + 射线拾取
- `survey` / 高密度：GPU picking pass

这是比“全 CPU”或“全 GPU”都更适合 v2 的方案：

- 漫游近景中，CPU 拾取更直接，也更容易和阅读交互结合
- `survey` 和高密度总览中，GPU picking 能显著降低命中测试成本，尤其在节点、标签和航迹同时存在时更稳定

### 8.6.10 数据传输格式

数据加载策略采用：

- 首屏 bootstrap 保持 JSON
- 后续邻域 / 视口增量块转向更轻量的列式或二进制格式

理由：

- 首屏 JSON 便于页面壳直接拼装与调试
- 进入宇宙后的高频增量数据，不应继续完全依赖冗长 JSON
- 因此正式主干应逐步切到更轻量的增量块表达，以压低解析与传输开销

### 8.6.11 音频运行时

音频运行时采用：

- `Web Audio`
- `AudioWorklet`
- 预渲染素材混音与空间控制结合

原因：

- 单纯主线程音频在高压交互场景里更容易受抖动影响
- `AudioWorklet` 更适合在运镜、聚焦、巡游切换时维持低延迟和更稳的空间化控制
- v2 不追求复杂音乐系统，但追求“静谧且稳定”的声场，因此需要更稳的底层控制能力

### 8.6.12 星球视觉生成时机

星球视觉资产采用：

- 构建期基础 atlas
- 运行时按 `seed` 惰性补齐变体

原则：

- 不把所有变体都在构建期烘死，避免素材体积过大
- 也不把全部视觉细节都推到运行时实时生成，避免首帧与切换时抖动
- 通过“基础 atlas + seed 惰性补齐”的方式，把资源体积、首屏速度与运行时灵活度控制在更稳的平衡点上

### 8.6.13 首屏启动时序

首屏启动必须围绕“先稳，再显现”来组织，而不是“先尽快出一个半成品画面”。

推荐启动时序：

1. 服务端页面壳返回：
   - locale
   - 开场文案候选
   - 支持环境说明
   - bootstrap 骨架数据
2. 主线程完成能力探测：
   - `Chrome 130+`
   - `WebGPU`
   - `Dedicated Worker`
   - `OffscreenCanvas`
3. 若不满足主干能力，立即进入不支持提示页
4. 若满足，则立即创建：
   - render worker
   - data worker
   - OffscreenCanvas 绑定
5. render worker 完成：
   - GPU adapter / device 请求
   - 基础 render graph 初始化
   - 开场星球所需 atlas / shader / 材质最小集准备
6. data worker 完成：
   - bootstrap 骨架解包
   - 开场星球邻域与 survey 基础结构装载
7. 只有当“最小可用首屏包”准备完毕，开场镜头才允许真正开始

原则：

- 首次进入允许存在短暂但稳定的预热时间
- 不接受“空白几帧后再慢慢补齐”的首帧抖动体验
- 用户第一次看到的宇宙必须已经是稳定构图，而不是尚在拼装中的施工现场

### 8.6.14 预热与资源门控

预热策略以“关键路径优先”组织：

- `P0`：
  - 开场星球
  - 首屏星尘背景
  - 开场运镜最小 shader / pipeline
  - HUD 极简层
- `P1`：
  - 当前焦点一跳邻域
  - `survey` 骨架
  - 首批标签 atlas
  - 首批路径高亮资源
- `P2`：
  - 二跳 / 三跳邻域
  - 巡游候选点预取
  - 次级特效
  - 帮助与教程的非关键静态资产

门控规则：

- 开场前只等待 `P0`
- 开场进行时后台加载 `P1`
- 用户第一次主动搜索 / 聚焦前，尽量完成 `P1`
- `P2` 只允许在空闲预算内推进

### 8.6.15 帧预算目标

性能目标必须以交互模式区分，而不是只写一个笼统的 FPS。

推荐运行时目标：

| 模式 | 目标 |
| --- | --- |
| `roaming` | 在支持环境的 `low` 档下，`1080p` 目标稳定 `60fps` |
| `focused` | 聚焦后仍保持 `60fps` 目标，允许局部质量提升但不得压垮输入响应 |
| `survey` | 优先保证缩放、平移、点选的连续性；不得因转场装饰拖慢操作 |
| `flying_cinematic` | 允许短时波动，但镜头不得产生明显掉帧和断续感 |
| `cruising` | 以平稳连续为核心，不追求高密度视觉炫技 |

预算原则：

- 相机更新、输入处理、命中测试属于最高优先级预算
- 标签、二三跳高亮、次级 halo、次级粒子属于可牺牲预算
- 当预算紧张时，先减画面复杂度，再减动画幅度，最后才考虑延迟非关键内容出现

### 8.6.16 运行时调度策略

运行时调度分为四条优先级通道：

1. `critical lane`
   - 输入
   - 相机
   - picking
   - 焦点切换
2. `visible lane`
   - 当前焦点
   - 一跳邻域
   - 当前路径高亮
   - 当前屏幕中央标签
3. `secondary lane`
   - 二跳 / 三跳邻域
   - 次级标签
   - 次级特效
   - 巡游远端候选点
4. `idle lane`
   - atlas 补齐
   - 未来巡游预取
   - 帮助面板静态资源
   - 调试与埋点整理

调度规则：

- 任意时刻 `critical lane` 可抢占其他通道
- 用户输入一旦发生，巡游与非关键预取必须立即让路
- `survey` 打开速度优先于花哨转场；必要时可以缩短、简化或跳过部分动画

### 8.6.17 飞行与静止状态的质量切换

为了保住性能，v2 必须接受“运动中先简化，停稳后再补细节”的策略。

具体规则：

- 飞行期间可临时：
  - 降低标签密度
  - 关闭二跳 / 三跳高亮
  - 降低次级 halo 与粒子强度
  - 推迟次级 `Spatial Card` 辅助信息展开
- 镜头停稳后再逐步补回：
  - 焦点标签
  - 一跳邻域标签
  - 次级信息抽屉可用性
  - 局部精细特效

这条规则同样适用于 `low` 档：

- `low` 档飞行时可以更激进地删减视觉负担
- 但不得牺牲焦点识别和一跳邻域理解能力

### 8.6.18 近景表达边界

近景高质量表达只保证到：

- 当前焦点星球
- 一跳邻域
- 当前主路径

二跳与三跳的原则：

- 可以存在
- 但以符号化、稀疏化、弱标签化表达为主
- 不追求“一次全部都画得很华丽”

这既是性能约束，也是认知约束：

- 用户在一次停留里最需要理解的是焦点与直接关联
- 更远层级的关系应等待用户主动继续探索

### 8.6.19 被明确排除的高成本效果

为了守住“性能第一”，以下效果默认不进入首版主干：

- 重型实时阴影
- 实时 `SSR`
- 实时 `SSGI`
- 高成本屏幕空间多级反射链
- 大范围体积云海式后处理
- 以高频全屏 pass 堆出的电影滤镜系统

电影感主要来源于：

- 镜头
- 构图
- 亮暗层次
- 航迹与路径表现
- 局部材质细节

而不是依赖昂贵的全屏特效堆叠

取舍：

- 当前 `3d-force-graph` 更适合 v1 / v1.5 阶段
- 如果要实现 v2 的电影化运镜、空间卡片、巡游、清屏、21:9、星体分层和更强控制，前台应逐步摆脱对通用图谱控件的依赖
- 因此前台重点不是“换框架”，而是“换图形内核控制方式”
- 同时，v2 的技术心智也从“尽可能兼容”切换为“在严格支持环境内做到最强体验”

## 9. 数据契约与接口升级

### 9.1 页面启动数据

统一页面 bootstrap 建议新增：

| 字段 | 说明 |
| --- | --- |
| `portal_mode` | 固定为 `universe_os` |
| `initial_intent` | `roam / search / entry / submit / tutorial` |
| `intro_copy` | 当前 locale 的开场文案模板 |
| `capabilities` | 当前实例可用能力 |

### 9.1.1 多语言显示原则

- 知识库词条标题默认显示原本语言
- 若条目存在 alias，可在搜索与辅助显示中使用 alias 作为补充入口
- 当需要同时显示标题与 alias 时，原始标题为主，alias 为副
- 操作界面文案才做中英文 locale 翻译
- 默认语言为英文；只有本地语言为中文时，才默认切到中文界面

### 9.2 图谱 payload

现有 graph payload 需要升级，使前台不仅能“画图”，还能“驱动宇宙操作系统”。

节点建议补充：

- `celestial_seed`
- `render_profile`
- `camera_anchor`
- `importance_rank`
- `detail_preview`
- `neighbor_ids`
- `highlight_group`
- `entry_route`

边建议补充：

- `strength`
- `trail_rank`
- `dim_group`
- `focus_group`
- `is_primary_route`

### 9.2.1 知识真相源与派生索引

v2 后端继续坚持：

- Markdown 文件是知识真相源
- Obsidian 文件结构、双链、frontmatter 与正文仍是核心工作流
- 宇宙前台只读取 **从 Markdown 派生出的只读索引**

推荐派生层：

- `entry index`
  - 标题、alias、语言、摘要、路径、mtime
- `graph index`
  - 节点、边、importance、邻域、稳定坐标
- `survey index`
  - 俯瞰态专用聚合点、主标签、骨架连线、热点簇
- `cruise index`
  - 巡游候选点、节奏权重、类别多样性权重、近期已访问排除信息

约束：

- 派生索引可以落地为缓存文件、内存对象或启动后生成的中间产物
- 但任何编辑、提交、审核、人工维护的最终归宿仍然是 Markdown
- 不引入与 Markdown 平行的“主数据库”作为宇宙前台的真实来源

### 9.2.2 图块切分策略

图数据切分采用 **知识盆地 + 视口** 的混合策略，而不是单一维度切块。

原因：

- 只按视口切块，容易破坏知识结构连续性
- 只按语义簇切块，俯瞰态与镜头移动时又容易出现空间分页割裂

因此推荐双层切块：

- 第一层：按 `Knowledge Basin / cluster_anchor` 做语义主分区
- 第二层：在每个 basin 内按空间视口进一步切成轻量 sector / tile

这样可以同时满足：

- 漫游与聚焦时保留语义邻域完整性
- `survey` 缩放和平移时维持更稳定的按视口加载
- 巡游时更容易做“在同一盆地内深一点，再跃迁到另一个盆地”的节奏组织

### 9.2.3 survey 专用索引

`survey` 不应复用漫游态的完整运行时数据，而应拥有轻量专用索引。

`survey index` 至少承担：

- 聚合后的可点选节点骨架
- 主标签候选
- 主干路径 / basin 边界
- 搜索命中映射坐标
- zoom level 对应的显示层级

原则：

- `survey` 是战略星图，不是把漫游态场景硬拉远
- 它需要的是更轻、更稳、更地图化的数据形态
- 因此前后端都应把 `survey` 视作独立查询面，而不是漫游态的附带模式

### 9.2.4 巡游候选池

巡游路线采用“后端预编排候选池 + 前端本地受约束随机”的混合策略。

后端负责：

- 预先计算值得巡游的候选点集合
- 给出稳定知识 / 形成中知识 / basin / navigation structure 的平衡权重
- 过滤不适合公开展示或当前不应前台强调的对象

前端负责：

- 在当前会话中做受约束随机
- 避免重复
- 根据用户最近看过的对象进行去重与节奏重排
- 在预算允许时对即将到来的候选点做轻量预取

这样做的好处是：

- 路线质量有后端知识语义托底
- 会话层的流动感和偶然性仍然由前端掌控
- 巡游不会既像死脚本，也不会像完全无脑乱跳

### 9.3 前台 API 族

建议引入一组新的前台宇宙接口，旧接口保留兼容：

- `GET /api/portal/universe/bootstrap`
- `GET /api/portal/universe/graph`
- `GET /api/portal/universe/graph/sector`
- `GET /api/portal/universe/node/{id}`
- `GET /api/portal/universe/search`
- `GET /api/portal/universe/survey`
- `GET /api/portal/universe/cruise/candidates`
- `POST /api/portal/universe/submit/text`
- `POST /api/portal/universe/submit/document`

### 9.3.1 接口职责建议

推荐职责切分如下：

| 接口 | 职责 |
| --- | --- |
| `bootstrap` | 返回首屏最小可用包、能力说明、开场数据、初始意图 |
| `graph` | 返回当前默认宇宙入口所需的骨架图信息 |
| `graph/sector` | 按 basin + 视口返回增量 sector / tile |
| `node/{id}` | 返回 `Spatial Card` 完整阅读所需内容与相关邻域 |
| `search` | 返回搜索命中骨架、命中类型、焦点建议、最小预览 |
| `survey` | 返回战略星图专用轻量索引 |
| `cruise/candidates` | 返回巡游候选池与排序权重 |

### 9.3.2 搜索返回策略

搜索接口默认采用“两段式返回”，优先保性能和反馈速度：

- 第一段：
  - 返回命中骨架
  - 返回命中坐标 / focus 建议
  - 返回标题、alias、最小摘要
- 第二段：
  - 用户真正选中目标后，再请求完整节点详情与阅读内容

原因：

- 搜索框的第一职责是帮助用户快速定位
- 不是在输入阶段就把完整 Markdown 和大邻域全部拉回前端
- 这样能让 `survey` 打开与搜索输入更轻、更稳、更像导航系统

兼容策略：

- 旧 `/api/portal/search*`、`/api/portal/graph*`、`/api/portal/submissions/*` 在迁移期继续存在
- 新前端不再直接消费旧接口

## 11. 验证方案

### 11.1 契约用例

建议新增 / 更新以下方向的测试：

- 前台所有主路由都返回同一宇宙壳
- `initial_intent` 与 URL 参数一致
- 搜索以内嵌 `survey` 导航态完成；详情、提交、教程以内嵌 overlay 形式完成
- 刷新页面、后退、直接访问深链都能恢复宇宙状态
- fixed coordinates 不随交互漂移
- reduced motion、调试 fallback 与不支持环境提示行为可预测
- 首次访问播放不可打断开场；回访默认跳过
- 搜索打开时进入全屏 `survey` 星图态，而不是普通输入框下拉
- 移动端访问时不进入宇宙交互，而是得到桌面端提示
- 触屏桌面端设备仍可完成核心探索、搜索与聚焦
- 标签在常态、选中态、`survey` 态下遵守优先级显隐规则
- `Spatial Card` 点击后出现完整 Markdown，且窗口受控并可滚动
- 浏览器阻止自动播放时，出现极轻的 `tap to enable sound` 提示
- 搜索结果列表停靠在左上搜索框下方
- 相关知识近距跳转时直接平移，远距跳转时进入电影化运镜
- HUD 默认处于极简态，可展开到完整功能态

### 11.2 E2E 场景

核心 E2E 用例建议锁定：

1. 首次进入首页即进入宇宙
2. 搜索远距节点触发电影化飞行
3. 点击节点后在宇宙内展开详情
4. 在宇宙内完成文本提交
5. 清屏后进入巡游并被人工操作中断
6. 不支持环境会被明确拦截；调试 fallback 场景下行为可预测
7. 20 秒无操作后自动进入巡游
8. 电影化运镜期间出现并退出 21:9 黑边
9. 选中节点后，未连接节点几乎隐去，只保留微弱背景星尘
10. 选中节点后，镜头尽量容纳一跳邻域
11. 路径高亮按重要度逐层衰减，且最多显示三跳
12. `survey` 中只能平移、缩放与点选，不能旋转
13. 巡游期间可使用 `Pause / Next / Exit`，且 `Exit` 后停留在原地

## 12. 已确认的决策

1. 前台完整宇宙化；不保留传统门户体验作为主路径
2. 后台不参与宇宙化改造
3. 搜索、结果、详情、提交、教程都优先在宇宙内完成
4. 首发渲染策略采用“混合分层”，以核显可跑为硬约束
5. 清屏巡游是一级能力，不是装饰性屏保
6. 节点视觉采用“类型骨架 + seed 变体”，并严格与 locale 文案解耦
7. 搜索的产品语义是“导航指令”，并与弹出式 3D 星图共用同一入口
8. 条目主交互采用贴附星球的 `Spatial Card`，并支持收缩
9. 首次访问必须播放 2 到 3 秒的高完成度开场镜头；回访可跳过
10. 页面 20 秒无操作自动进入巡游模式
11. 巡游镜头优先级为 沉浸展示 > 自动发现 > 导览说明
12. 前台优先保留 Python 页面壳，但图形运行时必须升级为自管高性能内核
13. 英文优先；视觉层尽量无语言，语言差异只落在 UI 与内容层
14. v2 首发不支持移动端；移动端直接提示使用电脑端访问
15. 声音只使用克制的音效，不使用旁白
16. 搜索以全屏俯瞰星图态承载，而不是局部弹窗或迷你地图
17. `Spatial Card` 固定在稳定阅读位，不跟随星球实时漂移
18. 用户提交不会立即在宇宙中生成可见星球
19. `low` / `high` 的区别主要在渲染精度，不在同屏节点规模
20. 默认宇宙配色采用深蓝冷黑 + 金色文明感
21. 选中节点后，未连接区域几乎隐去，只保留微弱背景星尘
22. `survey` 相机优先采用战略星图视角
23. `Spatial Card` 默认按目标位置智能切换左右
24. 标签默认少量高优先级显示，并按常态 / 选中 / `survey` 三种状态切换重点
25. 声音默认开启，角落提供常驻静音按钮
26. 巡游路线采用“稳定知识与形成中知识交替 + 受约束随机”的生成方式
27. 路径高亮最多显示三跳，并按重要度逐层衰减
28. 选中节点后的镜头应尽量纳入一跳邻域
29. 所有知识类型都需要对应天体表达，但前台感知类别可适度收束
30. 提交成功以“短暂出现又消散的种子光点”表达被系统接收
31. iPad 横屏与触屏 Windows 设备视作桌面端并纳入交互设计范围
32. 整体视觉强度采用“克制中带少量精致特效”
33. `survey` 只允许平移、缩放与点选，保持强地图感
34. `Spatial Card` 只在点击后出现，并承载完整 Markdown 阅读
35. 浏览器拦截自动播放时，以极轻的 `tap to enable sound` 提示兜底
36. 知识标题默认显示原本语言，界面文案才按 locale 翻译
37. 最低体验基线为 `720p`，最佳视觉目标为 `1080p`
38. 巡游概要靠近目标星球出现，但不贴附、不遮挡主焦点
39. 前 30 秒主叙事采用“史诗抵达型 + 任务导航型”的组合
40. 开场停留星球从稳定知识中按“代表性 + 可读性 + 视觉完成度”加权随机
41. 开场停留星球显示真实知识标题 + 一句真实知识摘要
42. 诗性开场文案来自约 100 条候选池，并在每次访问时随机抽取
43. HUD 默认极简常驻，可展开为完整功能态
44. 帮助面板优先级为：操作方式 > 搜索/Survey/巡游 > 天体类别 > 提交与教程 > 设置
45. Skill / MCP 介入教程必须保留在帮助/教程体系中
46. 搜索结果列表默认停靠在左上搜索框下方
47. 搜索覆盖标题、alias 与全文召回，但不把自然语言问答并入搜索框主语义
48. 在 `survey` 选中目标后立即退出并进入聚焦/电影化运镜
49. `survey` 采用中等节点密度，兼顾结构可读性与可点选性
50. `Spatial Card` 采用中等尺寸文档窗
51. `Spatial Card` 中所有信息都可查看，但次级信息默认用抽屉收起
52. 从 `Spatial Card` 点击相关知识时，近距直接平移，远距电影化运镜
53. 前台用户感知类别固定收束为 5 类：Stable Knowledge / Forming Knowledge / Question Constellations / Knowledge Basins / Navigation Structures
54. 系统内部类型名默认隐藏，只在帮助中解释
55. 巡游停留时长动态变化，默认约 3 秒
56. 巡游期间保留极简的 Pause / Next / Exit 控制
57. 巡游可停靠对象包括稳定知识、形成中知识、Knowledge Basins 与 Navigation Structures
58. 巡游诗性句来自模板，事实句来自真实知识摘要
59. 声音采用极轻环境底噪 + 事件音效的组合
60. 静音按钮固定在右上角
61. 新宇宙运行时模块统一使用 TypeScript
62. 音效实现采用“预渲染素材 + Web Audio 控制”的混合方案
63. 图形能力基线改为 WebGPU，正式支持环境按桌面端 Chrome 130+ 设计
64. 若缺少 WebGPU、Dedicated Worker 或 OffscreenCanvas 主干能力，则直接进入不支持提示，而不是提供残缺 fallback
65. v2 默认采用 Worker-first 渲染架构，由 Dedicated Worker + OffscreenCanvas 承载主场景循环
66. 文字标签采用 MSDF / SDF atlas 方案，不依赖 DOM 标签墙或临时 sprite text
67. 星球视觉采用 seed 驱动的程序化生成 + atlas / texture cache
68. 数据加载采用 bootstrap 骨架 + 邻域 / 视口渐进式加载
69. `survey` 与 `roaming` 共用同一场景内核，只切换相机 rig 与表达层
70. `high` 档允许引入少量高级后效，但必须保持克制与稳定
71. 性能观测采用 FPS + Long Animation Frames + 自定义镜头 / 加载埋点
72. v2 采用直接替换前台门户的上线策略
73. `720p` 下优先收紧 HUD 与阅读窗，并同步下调特效强度
74. v2 的技术优先级固定为：性能 > 质量 > 兼容
75. 渲染抽象层采用 `three/webgpu` 起步，并为热点模块保留原生 WebGPU 逃生口
76. 常规材质优先走 TSL / node material，关键热点特效允许手写 WGSL
77. Worker 拆分采用 render worker + data worker 的双 Worker 主结构
78. 主线程与 Worker 之间采用 command bus + snapshot state 通信模型
79. Picking 采用近景 CPU、俯瞰/高密度 GPU 的混合方案
80. 首屏维持 JSON bootstrap，后续增量图块转向更轻量的列式或二进制格式
81. 音频运行时采用 Web Audio + AudioWorklet
82. 星球视觉采用“构建期基础 atlas + 运行时按 seed 惰性补齐”的资产生成策略
83. 首次进入允许短暂预热，但禁止先显示半成品画面再补资源
84. 预热按 P0/P1/P2 分级组织，只等待开场关键路径资源
85. 运行时调度采用 critical / visible / secondary / idle 四通道
86. 飞行中允许临时削减标签、二三跳高亮和次级特效，停稳后再补
87. 近景高质量表达只保证焦点、一跳邻域与主路径
88. `survey` 打开速度优先于花哨转场，必要时可主动缩短动画
89. 首版默认排除重型实时阴影、SSR、SSGI 等高成本效果
90. Markdown 明文知识库继续是系统真相源，Obsidian 工作流继续保留为核心能力
91. Universe OS 面向人类用户，Agent 继续通过 MCP / Skill 接入
92. 如需高性能数据组织，只允许在 Markdown 之上构建派生索引 / 派生缓存
93. 图数据切分采用“Knowledge Basin + 视口”的混合策略
94. `survey` 使用专用轻量索引，不复用漫游态完整数据形态
95. 巡游采用“后端预编排候选池 + 前端受约束随机”的混合路线
96. 搜索接口默认采用“命中骨架先返回、完整详情后拉取”的两段式返回策略
