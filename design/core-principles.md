# Sediment 核心精神

这份文档只描述 Sediment 在多个版本中都应保持稳定的设计原则，不绑定某个具体条目模板或检索实现。

## 1. 系统目标

Sediment 要解决的不是“如何把原始材料全部收进来”，而是如何把组织中的隐性知识沉淀为一个可靠、可审阅、可协作的知识层。

它不是：

- 原始材料的镜像仓库
- 黑盒检索系统
- 只对最强模型有效的脆弱工作流

它追求的是：从文档、代码、复盘和流程记录中提炼可迁移的知识，并把这些知识保存在人和 Agent 都能共同操作的白盒介质中。

## 2. 不可妥协的原则

### 2.1 白盒优先

知识库首先是文件，而不是隐藏数据库。

- 人类必须能直接打开、编辑、审阅和迁移
- Agent、脚本与人工编辑应围绕同一份 Markdown 数据协作
- 如果一种设计明显提高了系统能力，却显著削弱了可见性和可迁移性，它默认不应被采用

### 2.2 Provenance 只是元数据

来源必须可追溯，但来源名默认不是知识图谱节点。

- 来源应以元数据形式保留
- 来源文档名、会议名、复盘标题、文件名默认不产生图谱边
- 只有当某个来源名称在领域中已经被反复当作一等概念使用时，才可以被提升为正式知识条目

### 2.3 中结构优先于过度自由

知识条目必须有足够结构，才能支撑稳定的整理、巡检和检索；但结构也不能重到把知识写成难以阅读的表单。

- 结构的职责是保护下限，而不是压制表达
- 模板应该优先帮助中等水平的 LLM 稳定工作
- 不允许为了追求机器可处理性，把知识正文变成人类难读的机械格式

### 2.4 人工审核是一等公民

Sediment 默认人类会介入审阅、修正和补充。

- 系统要鼓励人工直改，而不是把所有修改都包进黑盒流程
- Agent 生成的内容必须适合人工判断与接手
- “可审计”比“全自动”更重要

### 2.5 先保护知识层，再追求自动化

自动化不是目标，可靠性才是目标。

- 摄入阶段优先做首次提炼，不要过度优化
- 全局重构、结构修复、缺口提升应更多交给整理阶段
- 检索阶段应明确承认 gaps 与 contradictions，而不是用猜测掩盖证据不足

## 3. 设计演进的判断标准

如果未来要引入新的 schema、索引结构或自动化机制，至少要回答清楚下面四个问题：

1. 它是否仍然保持白盒与可人工编辑？
2. 它是否仍然把 provenance 和知识图谱分开？
3. 它是否真的提高了中等水平 LLM 的稳定下限？
4. 它是否让知识更易审阅，而不是更像黑盒数据库？

只有同时满足这些条件的变化，才值得进入“当前设计”。

## 4. 文档分层原则

核心精神和具体设计必须解耦：

- 这份文档定义“不变的约束”
- `design/current/` 定义“当前采用的实现组织方式”
- `design/evolution/` 记录“设计是如何一步步演进到这里的”

这样做的目的不是增加文档数量，而是避免每次演进都重写系统的根本叙事。
