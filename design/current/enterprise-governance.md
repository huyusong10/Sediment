# 当前设计：企业治理模型

当 Sediment 部署在企业环境中时，它不再只是一个本地知识库，而是一个由多人、多团队、多个工作流共同维护的分布式知识系统。

此时系统会同时面临两种失效模式：

- 如果只有少数负责人能把知识写入正式知识层，负责人会变成吞吐瓶颈，系统退化为中心化解释服务
- 如果任何人都能无约束地把内容直接写入正式知识层，知识库会被草稿、重复表述和局部真相腐蚀

因此，Sediment 的企业治理目标不是在“完全中心化”和“完全放开”之间二选一，而是建立一种联邦式治理：

- 允许广泛贡献
- 限制正式知识的提升权限
- 用结构化规则和巡检机制保护知识层

## 1. 治理目标

企业部署下的治理模型应同时满足五个目标：

- 让知识录入入口足够宽，避免“只能问负责人”
- 让正式知识层保持可审阅、可追溯、可纠错
- 让知识责任尽量下沉到业务域，而不是堆给中央团队
- 让 `health` / `tidy` 输出稳定治理队列，而不是依赖个人记忆维持质量
- 让 `knowledge_ask` 只面向当前已合并、可见、可审阅的知识状态

## 2. 基本判断

Sediment 在企业里应明确区分三件事：

- 贡献知识线索
- 整理知识提案
- 提升为正式知识

这三件事不应由同一权限无差别承担。

更具体地说：

- “允许提交材料”不等于“允许写入正式知识”
- “允许创建 placeholder”不等于“允许把 placeholder 直接升级为 `concept`”
- “允许回答问题”也不等于“允许绕过知识层直接从原始材料生成组织共识”

## 3. 联邦治理结构

推荐把治理结构拆成两层：

- 中央治理层：维护统一 schema、校验规则、索引规范、健康检查规则和运行时边界
- 领域治理层：按团队或业务域维护各自的知识条目、索引入口、整理节奏和审阅责任

中央治理层不负责吞掉所有知识编辑请求。

它的职责是：

- 定义哪些条目算合法
- 定义哪些质量问题必须进入治理队列
- 保持跨团队知识层的一致下限
- 处理跨域冲突和规则演进

领域治理层才是正式知识的主要维护者。

它的职责是：

- 决定本域哪些内容值得进入正式知识层
- 审阅 `ingest` 产出的条目提案
- 处理本域 `health` 报告中的硬错误和高价值告警
- 维护本域索引与知识入口

## 4. 角色分层

### 4.1 Anonymous Contributor

本阶段不单独引入“已认证 contributor”角色。

公开门户默认使用匿名贡献模型，任何知识生产者都可以不登录直接提交材料。

可以做的事：

- 提交原始材料、复盘、会议纪要、流程说明、代码线索
- 提交条目修改提案
- 创建或补充 placeholder
- 对现有条目提出修正建议

系统行为：

- 默认记录 `submitter_name` 与可信来源 IP
- 当提交者同时携带有效 owner / committer session 时，额外记录 `submitter_user_id`

不应直接拥有的权力：

- 无审阅地把内容写入正式知识层
- 无审阅地把 `placeholder` 提升为 `concept` / `lesson`
- 无审阅地修改全局索引和通用规则

### 4.2 Committer

这里的 `committer` 指“有权把提案提升进正式知识层的人”，不是狭义的 Git commit 执行者。

领域 committer 是本域正式知识层的第一责任人。

可以做的事：

- 审阅并合并本域条目提案
- 判断某个内容应该成为 `concept`、`lesson` 还是 `placeholder`
- 决定 `status` 应为 `fact`、`inferred` 还是 `disputed`
- 处理本域 `health` / `tidy` 工作队列
- 维护本域索引入口与条目连接质量

核心要求：

- committer 负责提升质量，不负责垄断输入
- committer 应服务于吞吐，而不是成为“唯一会写知识的人”

### 4.3 Owner / Platform Maintainer

`owner` 负责 Sediment 机制本身，而不是替各领域写知识。

可以做的事：

- 维护条目模型、校验器、健康检查和运行时接口
- 管理默认分支保护、自动检查和知识库发布方式
- 管理用户、展示/停用 token、查看系统状态
- 构建 Quartz、处理服务器级设置
- 处理跨域结构问题和规则升级

不应长期承担的事：

- 为所有团队代写和代审知识条目
- 成为跨企业唯一知识审批点

## 5. 权限模型

企业部署建议把权限设计成“开放贡献，受控提升”。

最低可行权限模型如下：

- 所有人可 `ask`、`list`、`read`
- 所有人可提交材料和条目提案
- 领域 committer 可合并本域正式知识
- `owner` 可修改 schema、校验规则和全局运行时约束

本阶段实际落地的认证角色仅有：

- `owner`
- `committer`

其余公开用户统一视为 `anonymous`，不创建单独的长期 contributor token。

补充约束：

- owner 账户在一个实例中有且只有一个
- owner 由 `sediment init` 生成，并作为长期平台维护身份保留
- 后续 UI / CLI 只能新增 committer，不能再派生第二个 owner

如果采用 Git 作为知识层承载介质，推荐对应到如下机制：

- 默认分支承载当前 canonical knowledge state
- 默认分支受保护，不允许普通贡献者直接写入
- 领域目录或索引段由对应 committer 审阅
- 结构校验和 `health` 检查作为合并前质量门槛

这意味着企业环境中真正开放的应当是“提案入口”，而不是“直接改 canonical branch 的权力”。

## 6. 提交流向

推荐的知识流动路径如下：

1. 贡献者提交原始材料或修订建议
2. `ingest` 生成“足够好的首次提炼”
3. 对不清晰、证据不足或仅被引用的概念，优先落为 `placeholder`
4. 领域 committer 审阅提案，决定是否合并为正式条目
5. 已合并知识进入 `knowledge_ask` 的可见范围
6. `health` 持续输出治理队列，`tidy` 负责长期结构修复和提升

这条路径的关键判断是：

- 宽入口放在步骤 1 和 2
- 强约束放在步骤 4 和 6
- 运行时只读取步骤 5 的结果

## 7. 条目生命周期

Sediment 现有条目模型已经足以支撑企业治理，不必一开始就引入更重的审批 schema。

推荐的生命周期如下：

- 未形成稳定结论：保留在原始材料或提案阶段
- 已知存在但定义不足：进入 `placeholder`
- 能形成可迁移判断但证据仍有限：进入 `lesson` 或 `concept`，并标记 `status: inferred`
- 已被多次验证、边界稳定：提升为 `status: fact`
- 存在争议或版本差异：标记 `status: disputed`

这里的重点不是“所有东西都尽快变成 fact”，而是让不确定性以结构化方式显式存在。

## 8. 正式知识的质量门槛

正式知识层至少应满足当前设计中的硬约束：

- formal entry 的 `type`、`status`、`sources` 合法
- `concept` 有实质性的 `Scope`
- `lesson` 有实质性的 `Trigger`、`Why`、`Risks`
- `Related` 至少包含一个真实知识链接

在企业治理里，还应增加三条审阅判断：

- 证据不足时，优先合并为 `placeholder` 或 `inferred`，不要强行写成 `fact`
- 某条正式结论如果只靠单个含混来源支撑，不应被提升为高置信公共知识
- 如果改动会改变多个团队的操作边界，应触发跨域复核，而不是只由单团队拍板

## 9. 领域边界与所有权

Sediment 不适合由一个中央编辑小组长期手工维护整个企业知识层。

更合理的方式是：

- 以业务域、平台域、基础设施域或职能域划分知识治理单元
- 每个单元维护自己的索引入口和整理节奏
- 每个正式条目都应能追溯到某个责任域
- 跨域条目由主责任域维护，相关域协同审阅

一个简单规则是：

- 单域知识由单域 committer 决策
- 跨域边界知识至少需要双边确认
- 全局规则只由平台层定义，不由某个业务域单独发明

## 10. 审阅与 SLA

企业部署时，真正需要治理的是等待时间，而不是形式上的审批动作。

推荐最小 SLA：

- 新材料或新提案在 1 个工作日内被归类
- 高优先级复盘知识在 3 个工作日内进入 `placeholder` 或正式条目
- `health` 硬错误按周清零
- 高引用 placeholder 按周进入 KB-level tidy 提升队列

这里“归类”比“写完美答案”更重要。即使暂时不能提升为正式知识，也应该先把它放进可追踪的缺口状态，而不是停留在某个人的私聊或脑海里。

## 11. 运行机制

推荐把日常治理节奏做成固定循环：

- 日常：自动运行 `health`，输出本域问题队列
- 每周：领域 committer 处理硬错误、断链、孤岛和高引用 placeholder
- 每双周或每月：做一次跨域索引和边界检查
- 重大事故后：要求相关领域在固定时间窗内完成一次 ingest 和最小知识沉淀

这样治理工作会从“临时想起来再修”变成稳定例行机制。

## 12. 关键指标

企业部署下，建议持续观察以下指标：

- 从材料提交到进入知识层的中位耗时
- 本域待审提案积压量
- `health` 硬错误数量
- dangling links、orphan entries、canonical gaps 的变化趋势
- 高引用 placeholder 的存量和平均停留时长
- `status: disputed` 条目的积压情况

如果后续运行时具备观测能力，还可以补充：

- `knowledge_ask` 返回 `gaps` / `contradictions` 的频率
- 高价值问题无法回答或只得到低置信回答的比例

## 13. 反模式

以下做法会让企业部署快速失效：

- 把所有合并权限集中在一个“知识管理员”身上
- 允许任何人绕过审阅直接改 canonical knowledge state
- 为了追求零告警，把条目模板变成难以书写的重表单
- 把来源名、会议名、文档名直接当作知识图谱节点扩图
- 因为证据不完整，就拒绝记录 placeholder 和 `inferred` 结论
- 让运行时直接基于未审阅原始材料生成组织共识

## 14. 推荐结论

Sediment 在企业里最适合的不是“中央审批制”，也不是“自由编辑制”，而是“联邦治理制”：

- 人人可贡献
- 领域可自治
- 正式知识受控提升

补充约束：

- owner token 由 `sediment init` 一次性生成并写入配置
- `sediment server start|run` 不再生成临时 admin token
- 审计日志始终记录服务端推导出的 `actor_id / actor_name / actor_role`
- 全局规则统一守底线

一句话总结就是：

不是所有人都能直接改正式知识层，但所有人都必须能把知识线索稳定送进系统，而且这些线索不能以负责人个人吞吐为前提。
