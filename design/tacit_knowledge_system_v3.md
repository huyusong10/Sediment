# 隐性知识提取系统设计文档

> 版本：v4.0 | 状态：设计阶段
> 说明：该文档已被 `design/tacit_knowledge_system_v4.md` 取代，现仅保留作历史参考。

---

## 1. 系统定位

企业中存在大量无法被人工描述、却客观存在于实践行为中的知识。本系统的目标是：

**将散落在文档、代码、流程记录中的隐性知识，自动提炼为结构化的原子知识条目，存储于可人工审核的白盒知识库中，供 Agent 在执行任务时主动检索使用。**

系统不追求自动化程度最高，而追求：**简单、可信、可维护**。

---

## 2. 核心用例

从使用者视角出发，系统支持以下五类用例：

### UC-1：批量摄入文档

**场景**：知识管理员将一批企业文档（设计文档、故障复盘、代码注释、会议记录等）导入系统，由 Agent 自动提炼知识。

**用户行为**：
```
管理员 → 提供文档目录或文件列表
       → 在 OpenCode 中触发摄入 Skill
       → 等待 Agent 处理
       → 在 Obsidian 中看到新增的 .md 文件
```

**实现方式**：OpenCode Skill（UC1-ingest）

---

### UC-2：人工触发整理

**场景**：知识库积累了一批原始条目后，知识管理员发现知识库质量下降（红链、重复条目、孤立节点、来源矛盾等），手动触发整理。

**用户行为**：
```
管理员 → 观察 Obsidian 图谱或健康报告，发现知识库质量下降
       → 在 OpenCode 中触发整理 Skill
       → 审核 Agent 提出的合并/拆分/补链/消解矛盾建议
       → 确认或拒绝
```

**实现方式**：OpenCode Skill（UC2-tidy）+ Python 辅助脚本（实时扫描文件系统，无状态）

---

### UC-3：Agent 检索知识

**场景**：用户 Agent 在执行任务时，需要查询本企业的专属知识。

**用户行为**：
```
用户 → 向 Agent 提出任务或问题
Agent → 调用 MCP 工具检索知识库
      → 将知识注入上下文
      → 完成任务
```

**实现方式**：MCP Server（knowledge_ask / knowledge_list / knowledge_read）

---

### UC-4：人工直接编辑知识库

**场景**：领域专家发现某条自动生成的知识条目有误，或想手动补充一条重要经验。

**用户行为**：
```
管理员 → 在 Obsidian 中直接打开 .md 文件
       → 修改内容、补充链接、调整表述
       → 保存即生效，下次整理时自动感知变更
```

**实现方式**：Obsidian 直接操作文件系统，无需任何系统介入

---

### UC-5：知识库健康巡检

**场景**：知识管理员想了解当前知识库的整体状况。

**用户行为**：
```
管理员 → 执行巡检脚本
       → 获得纯文本健康报告
       → 决定是否触发整理
```

**实现方式**：Python 脚本（health_check.py）统计指标，Skill 可选解读报告

## 4. 核心概念定义

### 4.1 原子知识条目（Atomic Knowledge Entry）

知识库的基本单元。**一个文件 = 一个可独立表述的命题或一组因果紧密相关的命题群**，不可再拆分为更小的有意义单元。

原子性原则：不需要拆分得特别细。当多个命题因果紧密（A 的结论是 B 的前提）时，允许合并在一个文件中，用段落自然衔接。后续有新素材加入时，整理阶段可将合并的知识拆分重组，以适应新的知识结构。

文件命名：文件名本身就是命题的标题，是 Agent 检索时的主要索引。命名要使用 Agent 在推理时自然会想到的词汇。

**条目模板（经验教训类）**：

适用于从事件、故障、复盘中提炼的规律性知识。

```markdown
# [命题标题]

> **核心**：[用 1-3 句话概括核心结论]
> **状态**：[事实 | 推断]

[推理过程：用自由文本叙述得出结论的过程、观察到的证据、以及关键权衡。
相关概念以 [[链接]] 形式自然嵌入正文中，每处链接都要有上下文说明
"为什么提到它"，而不是在底部集中列出。

反面案例或失败教训也在正文中叙述，让读者理解"不这样做会怎样"。]

## 上下文（可选）
- **适用场景**：[该知识在什么场景下适用]
- **前提条件**：[使用此知识需要已知什么]
- **不适用场景**：[该知识在什么情况下不适用]
- **环境假设**：[基于什么技术栈、组织架构、业务阶段的假设]

## 相关
- [[条目名]] — 一句话说明关系
- [[条目名]] — 一句话说明关系

## 来源
[[来源文档1]] [[来源文档2]]
```

**条目模板（概念/规范类）**：

适用于术语定义、技术规范、行为准则等不需要推理过程的条目。

```markdown
# [概念或规范名称]

> **核心**：[用一句话给出定义或规则]
> **状态**：事实

[补充说明：对该定义或规则的进一步解释、示例、或注意事项。
相关概念以 [[链接]] 形式自然嵌入正文中。]

## 上下文（可选）
- **适用场景**：[...
- **不适用场景**：[...

## 相关
- [[条目名]] — 一句话说明关系

## 来源
[[来源文档1]]
```

**状态标签定义**：

| 状态 | 含义 | 产生阶段 |
|------|------|---------|
| **事实** | 原文有明确表述，直接提炼 | ingest |
| **推断** | 从多份文档归纳推理得出，或 Agent 基于证据自行推断 | tidy（归纳推理）或 ingest 中 Agent 推断 |
| **存疑** | 多份来源对同一概念有矛盾解释，尚未解决 | tidy（发现冲突时创建，ingest 阶段不产生） |

**示例（经验教训类）**：

```markdown
# 多角色接口优先澄清权限边界

> **核心**：当接口设计涉及多个角色时，应在讨论功能细节之前先明确
> 各角色的权限边界，否则后期因权限理解偏差导致返工的成本远超前期沟通成本。
> **状态**：推断

在2024年Q3的支付模块故障复盘中发现，开发A认为"查看"接口不
需要权限校验，开发B认为需要登录态，测试C按无权限场景测。
根本原因是评审时没有人先明确"谁能调这个接口"，就直接进入
"返回什么字段"的讨论。

这里的核心是权限边界的定义顺序问题——必须先有[[权限模型]]，
才能讨论接口层面的实现。这也解释了为什么后续的
[[权限系统重构设计文档]]中，第一步就是建立角色-资源映射表。

反面案例：因权限理解偏差，上线后发现未登录用户可调用订单查询
接口，暴露其他用户的敏感信息，紧急回滚并修复耗时6小时。

如果走敏捷路线，有人可能会说"先实现功能再补权限"。[[敏捷开发规范]]
确实提倡先跑通核心路径，但多角色系统的经验表明：权限是**结构性
决策**，不像样式或文案那样可以后期迭代——它影响数据流向、接口
设计和测试策略的方方面面。

## 上下文
- **适用场景**：需求评审、接口设计评审阶段
- **前提条件**：系统存在多角色访问控制需求
- **不适用场景**：单角色内部接口、已有明确权限规范的成熟团队
- **环境假设**：RBAC 或 ABAC 权限模型

## 相关
- [[权限模型]] — 本知识的前置依赖，必须先有权限模型才能讨论接口实现
- [[接口设计规范]] — 本知识的具体应用场景
- [[敏捷开发规范]] — 存在张力：敏捷主张先跑通功能，本知识主张先定权限边界

## 来源
[[2024年Q3支付模块故障复盘]] [[权限系统重构设计文档]]
```

**示例（概念类）**：

```markdown
# 幂等性

> **核心**：同一操作执行一次或多次，产生的结果相同。
> **状态**：事实

在 RESTful API 设计中，GET、PUT、DELETE 方法天然幂等，
而 POST 方法不保证幂等。对于需要重试机制的接口（如支付下单），
必须通过唯一请求 ID 或业务主键来保证操作的幂等性。这在
[[接口幂等性设计]]中有具体的实现规范。

## 上下文
- **适用场景**：接口设计、消息队列处理、分布式系统
- **不适用场景**：日志记录、计数器等天然需要累积效果的场景

## 相关
- [[接口幂等性设计]] — 幂等性在接口设计中的具体应用
- [[重试机制]] — 需要幂等性作为前置保障

### 4.2 占位文件（Placeholder Entry）

当摄入阶段遇到一个被引用但尚无解释的概念时，系统自动创建的最小文件。它保证了知识网络的连通性——Agent 遇到这个概念时不会碰壁，而是知道"这里有个待填充的缺口"。

占位文件有两种来源：
- **悬空引用**：文档中出现了一个专有名词，但文档本身没有解释它
- **隐式概念**：多篇文档都在使用某个概念，但没有任何文档正式定义它

```markdown
# 幂等性

> 状态：占位（待填充）
> 出现于：[[支付接口规范]] [[订单系统设计]] [[消息队列使用规范]]

该概念在多篇文档中被引用，但尚未形成正式定义。
建议在下次整理时执行归纳推理。
```

被引用次数由整理脚本实时统计，不存储在文件中。

---

### 4.3 候选链接（Candidate Link）

摄入阶段，Agent 在知识条目内容中识别到的、意图引用但尚未确认目标是否存在的链接。用双括号标记：`[[概念名]]`。

候选链接在 Obsidian 中自动渲染为红链（目标不存在）或绿链（目标存在），无需额外标记状态。整理阶段通过实时扫描处理所有红链。

---

### 4.4 摄入（Ingest）

将原始文档转化为知识条目的过程，由 OpenCode Skill（UC1-ingest）驱动 Agent 执行。

摄入遵循**提取优先原则**：不在摄入阶段过度优化粒度，只负责将文档中的知识提炼为结构化的条目，保留推理过程和证据链。后续有新素材加入或知识网络演化时，由整理阶段负责拆分重组。

#### 4.4.1 压缩原则

知识库的价值不在于事无巨细地记录所有信息，而在于**压缩和提炼**。摄入时必须判断：

- **这是知识还是数据？**
  - 知识 = 可迁移的规律、经验、原则（"必须先校验库存，否则超卖"）
  - 数据 = 具体的值、实现细节、结构元素（"max_connections=100"、"def create_order()"）

- **这条知识是否已经存在于知识库中？**
  - 如果已有条目表述了相同或近似的规律，不产生新条目
  - 如果是对已有条目的补充或修正，由 tidy 阶段合并处理

如果输入全文没有可提炼的知识（如纯数据表、无注释的源码），则不产生任何条目。

#### 4.4.2 来源类型识别

不同类型的输入，提取策略不同：

| 输入类型 | 提取目标 | 过滤目标 |
|---------|---------|---------|
| **设计文档/复盘报告** | 结论、决策依据、教训、权衡分析 | 具体人名、会议时间、版本号 |
| **代码文件** | 注释/docstring 中的经验教训、代码结构暗示的设计决策（如先校验再执行的模式） | 函数名、变量名、类名、实现逻辑本身 |
| **API/接口文档** | 接口设计规范、权限约束、错误处理原则 | 单个接口的路径、参数列表、返回值 |
| **配置/部署文档** | 环境搭建规律、依赖关系、运维经验 | 具体配置项的值 |
| **表格/数据报表** | 从数据中呈现的模式（如"所有写操作延迟都高于读操作"） | 表格中的具体数据行 |
| **会议记录** | 决策结论及其理由 | 谁说的、讨论过程的流水账 |

#### 4.4.3 过滤规则（负向清单）

以下内容**绝对不应**作为知识条目：

- 变量名、函数名、类名、文件名
- 配置项的键名和具体值
- 单个接口的签名（路径、参数、返回值）
- 数据表中的具体数值行
- 人名、时间、会议名
- 没有规律性的事实（如"张三负责订单模块"——除非它揭示了一个组织模式）
- 纯描述性信息（如"系统使用了 Redis"——除非它揭示了"为什么选 Redis 而非 Memcached"的决策过程）

#### 4.4.4 正向案例

**代码输入**：
```python
def create_order(user_id, items):
    """
    创建订单。
    注意：必须先校验库存，再创建订单，
    否则会创建超卖订单。
    """
    check_inventory(items)
    order = Order(user_id=user_id)
    ...
```
应提取为：
```markdown
# 创建订单前必须先校验库存

> **核心**：创建订单时必须在创建动作之前校验库存，
> 否则会导致超卖。
> **状态**：事实

在 create_order 的实现中，check_inventory() 调用
位于 Order 创建之前，顺序不可颠倒。这是因为库存
校验是创建订单的前置条件（[[库存检查]]），而非后续补偿操作。

## 相关
- [[库存检查]] — 本知识涉及的具体校验逻辑
- [[订单系统设计]] — 订单创建流程的上层规范

## 来源
[[orders.py]]
```

**表格输入**（好的情况）：
```
| 接口 | 类型 | P99延迟 | QPS |
|------|------|---------|-----|
| /order | 写 | 120ms   | 500 |
| /user  | 读 | 50ms    | 800 |
| /pay   | 写 | 200ms   | 300 |
| /cache | 读 | 10ms    | 5000|
```
可提取为：
```markdown
# 写接口延迟显著高于读接口

> **核心**：该系统的写操作 P99 延迟（120-200ms）
> 显著高于读操作（10-50ms），优化时应区分对待。
> **状态**：事实

从性能数据可以看出，写操作的延迟是读操作的 2-20 倍。
在设置超时阈值和重试策略时（[[重试机制]]），写接口
需要更宽松的超时配置。

## 相关
- [[重试机制]] — 写接口的重试间隔需要更大
- [[接口超时配置]] — 不同操作类型应设置不同超时

## 来源
[[api-performance-report.xlsx]]
```

**配置输入**：
```yaml
# 连接池大小基于预估峰值 QPS 的 2 倍设置
# 低于此值会导致连接耗尽，高于此值无性能提升
redis:
  max_connections: 100  # QPS 峰值 50，2倍 = 100
```
应提取为：
```markdown
# Redis连接池按峰值QPS的2倍设置

> **核心**：Redis 连接池大小应设置为预估峰值 QPS 的 2 倍，
> 低于此值会导致连接耗尽，高于此值无性能收益。
> **状态**：事实

配置注释明确说明了此计算方式。连接池不足是此前
[[连接耗尽故障]] 的根因。

## 相关
- [[连接耗尽故障]] — 连接池设置过低的后果
- [[Redis配置规范]] — 更完整的 Redis 配置指南

## 来源
[[redis-config.yml]]
```

#### 4.4.5 负向案例

**代码——提取了实现细节而非知识**：
```
错误：
# create_order 函数

> **核心**：创建一个订单，包含 user_id 和 items 参数。

在 orders.py 第 42 行定义了 create_order 函数，
它接收 user_id 和 items 两个参数...

## 来源
[[orders.py]]
```
错误原因：把函数签名当知识。代码本身已经表达了函数签名，
不需要再用自然语言复述。

**表格——提取了数据行而非模式**：
```
错误：
# order接口的QPS是500

> **核心**：/order 接口的 QPS 为 500。

## 来源
[[api-performance-report.xlsx]]
```
错误原因：单个数据点不是知识。只有从多组数据中发现的模式
（如"写接口延迟高于读接口"）才是。

**配置——提取了配置值而非规律**：
```
错误：
# Redis最大连接数是100

> **核心**：Redis 的 max_connections 设置为 100。

## 来源
[[redis-config.yml]]
```
错误原因：配置值本身不是知识。配置值背后的**设置原则**才是。

**纯描述——没有可迁移的规律**：
```
错误：
# 张三负责订单模块

> **核心**：订单模块由张三负责开发。

## 来源
[[项目分工会议纪要]]
```
错误原因：组织分工的具体分配不是知识。但如果是"订单模块
由专门的团队维护，而非功能团队"——这可能揭示了一个组织模式，
可以提炼。
```

---

### 4.5 整理（Tidy）

对现有知识库状态进行梳理和重构的动作，**由人工主动触发，不自动执行**。整理完全无状态，所有判断依据均实时从文件系统计算。

整理不是只增不减的修补，而是**可以重构已有知识**的过程。当新材料加入或知识网络演化时，整理阶段可能拆分、合并、改写已有条目。

整理包含以下子动作，可按需选择：

**子动作 1：确认悬空链接**
实时 grep 所有条目中的 `[[链接]]`，与 `ls entries/ placeholders/` 的结果对比，对不存在对应文件的链接创建占位文件。

**子动作 2：归纳推理（侦探模式）**
实时统计各占位文件被引用次数（grep 计数），对高引用占位文件聚合所有引用上下文，由 LLM 推断含义，生成正式条目草稿供人工审核。

**子动作 3：合并重复条目**
对所有正式条目做语义相似度扫描，提出合并建议，由人工决定是否合并。

**子动作 4：拆分过粗条目**
当知识网络增长时，原本合并在一起的命题可能需要拆分为更细的原子条目以方便新的知识精确链接。由 LLM 分析条目内容，提出拆分建议。

**子动作 5：消解来源矛盾**
扫描全库，检测对同一概念给出矛盾解释的来源，将条目状态标记为"存疑"并创建"争议"区，列出矛盾观点及各自来源。分析矛盾是否可以调和（如适用场景不同），可以调和则在条目中澄清边界，无法调和则保留争议。

**子动作 6：升级知识状态**
将"推断"升级为"事实"（新来源确认了推断），或降级为"存疑"（新来源与推断矛盾）。

**子动作 7：补充孤立节点**
检测无任何入链或出链的条目，提示可能被遗漏，建议补充关联。

---

### 4.6 检索（Retrieve）

Agent 在执行任务时主动查找相关知识的过程。检索有两种路径，由调用方选择。

**路径 A：knowledge_ask（主力，推荐）**

适合模糊语义问题，内部由子 Agent 多轮推理：

```
接收用户问题
  ↓
子 Agent 调用 knowledge_list() 获取所有文件名
  ↓
LLM 推理出候选文件名列表
  ↓
调用 knowledge_read() 逐个读取
  ↓
顺着正文中嵌入的 [[链接]] 深入跳转
  ↓
必要时换关键词重新推理
  ↓
综合所有相关条目，返回结构化答案 + 引用来源列表
```

**路径 B：Skill + knowledge_list/read（自主探索）**

适合用户 Agent 自己想控制探索过程，或对 knowledge_ask 的结果存疑需要验证。Skill 描述探索协议，Agent 自主调用 list 和 read 完成检索。

两条路径使用同一套底层逻辑，区别在于执行主体：路径 A 由服务端子 Agent 执行，路径 B 由调用方 Agent 自己执行。

### 4.7 巡检（Health Check）

对知识库整体状态的快速扫描，输出健康报告。由 Python 脚本负责指标统计，可选由 LLM 解读报告并建议下一步行动。

```
知识库健康报告
==============
总条目数：342
  - 正式条目：289
  - 占位文件：53

占位文件引用分布：
  - 高引用（>=5次）：8个  ← 建议优先归纳
  - 中引用（2-4次）：21个
  - 低引用（1次）：24个

孤立节点（无任何链接）：17个
红链数量（目标不存在的链接）：31个

建议：执行一次整理，重点处理8个高引用占位文件。
```

## 6. 工具层设计

### 6.1 OpenCode Skill：UC1-ingest（摄入）

**触发方式**：管理员在 OpenCode 中手动调用

**Skill 内容概要**：
```
你是一个知识摄入 Agent。
目标：将给定文档提炼为原子知识条目，存入知识库。

知识库路径：/knowledge-base/
- 正式条目存入：entries/
- 占位文件存入：placeholders/

压缩原则：
知识库是压缩层，不是镜像。只提取可迁移的规律、经验、原则，
不记录具体的实现细节、数据、结构元素。
如果输入全文没有可提炼的知识，则不产生任何条目。

来源类型识别：
- 代码：只看注释/docstring 中的经验教训和设计决策暗示
  不提取：函数名、变量名、类名、实现逻辑本身
- 表格/数据报表：提取从数据中呈现的模式
  不提取：单个数据行、具体数值
- 配置/部署文档：提取环境搭建规律、依赖关系、运维经验
  不提取：具体配置项的值
- 设计文档/复盘报告：提取结论、决策依据、教训、权衡分析
  不提取：人名、会议时间、版本号
- API/接口文档：提取接口设计规范、权限约束、错误处理原则
  不提取：单个接口的路径、参数列表、返回值
- 会议记录：提取决策结论及其理由
  不提取：讨论过程的流水账、谁说了什么

负向清单（绝对不提取）：
- 变量名、函数名、类名、文件名
- 配置项的键名和具体值
- 单个接口的签名（路径、参数、返回值）
- 数据表中的具体数值行
- 人名、时间、会议名
- 没有规律性的事实（如"张三负责订单模块"）
- 纯描述性信息（如"系统使用了 Redis"）

条目生成规则：
- 一个命题 = 一个文件，因果紧密的命题群可合并
- 文件名使用自然语言概念名或"场景+结论"短语
- 根据内容类型选择合适的模板：
  · 经验教训类（从事件、故障、复盘中提炼）：
    - 必须包含：核心结论（1-3句）+ 推理过程（自由文本）+ 来源
    - 状态通常为"推断"
  · 概念/规范类（术语定义、技术规范、行为准则）：
    - 必须包含：核心定义（一句话）+ 补充说明 + 来源
    - 不需要推理过程
    - 状态为"事实"
- 相关概念以 [[链接]] 形式自然嵌入正文中，每个链接都要有上下文
  说明"为什么提到它"
- 在"## 相关"区提供轻量级快速索引，每条格式：
  [[条目名]] — 一句话说明关系
- 包含反面案例或失败教训（如有），让读者理解"不这样做会怎样"
- 无法在文档内解释的概念，在 placeholders/ 创建占位文件

状态判断：
- 事实：原文有明确表述，直接提炼
- 推断：从多份文档归纳得出，或基于证据自行推断
（"存疑"状态由整理阶段在发现来源矛盾时创建，ingest 阶段不产生）

处理流程：
1. 读取文档，识别来源类型（代码/表格/文档/配置等）
2. 根据来源类型应用对应的提取策略和过滤规则
3. 识别所有可独立表述的命题或命题群
4. 判断内容类型（经验教训 / 概念规范），选择对应模板
5. 判断状态标签（事实/推断）
6. 逐个生成 .md 文件
```

### 6.2 OpenCode Skill：UC2-tidy（整理）

**触发方式**：管理员在 OpenCode 中手动调用

**依赖脚本**：`tidy_utils.py`，提供以下无状态工具函数：

```python
# 实时扫描所有候选链接，返回红链列表（目标文件不存在的链接）
def find_dangling_links(kb_path) -> list[dict]

# 统计各占位文件被引用次数，返回排序列表
def count_placeholder_refs(kb_path) -> list[dict]

# 扫描无任何入链或出链的条目，返回孤立节点列表
def find_orphan_entries(kb_path) -> list[str]

# 提取指定占位文件的所有引用上下文片段
def collect_ref_contexts(kb_path, placeholder_name) -> list[str]
```

**Skill 内容概要**：
```
你是一个知识整理 Agent。
目标：梳理和重构知识库的内部一致性，不产生新的文档摄入。

可用工具：tidy_utils.py 中的函数（通过 Python 调用）

整理子动作（可按需选择）：
1. 确认悬空链接：调用 find_dangling_links()，为每个红链创建占位文件
2. 归纳推理：调用 count_placeholder_refs() 找高引用占位文件，
             再调用 collect_ref_contexts() 聚合上下文，
             推断概念含义，生成正式条目草稿（状态：推断），标记"待人工确认"
3. 合并重复：对 entries/ 下所有条目做语义相似性分析，提出合并建议
4. 拆分过粗：当知识库增长后，分析原本合并的条目是否需要拆分为更细的
             原子单元以方便新知识精确链接
5. 消解矛盾：扫描全库检测对同一概念有矛盾解释的来源，将条目状态标记为
             "存疑"并创建"争议"区。分析矛盾是否可以调和（如适用场景不同
             则各自澄清边界），无法调和则保留争议
6. 升级状态：有新来源确认了"推断"则升级为"事实"，
             新来源与已有"事实"矛盾则降级为"存疑"
7. 补充孤立：调用 find_orphan_entries()，建议为孤立节点补充关联
8. 补全索引：检查条目是否缺少"## 相关"区，建议补充轻量级快速索引

每个建议须经人工确认后再写入文件。
```

### 6.3 MCP Server 工具

MCP Server 部署在服务器上，对外暴露三个工具。

#### knowledge_ask(question: str) → str

**适用场景**：用户有模糊语义问题，想要综合答案。

**内部流程**：
```
接收 question
  ↓
启动 OpenCode CLI 子进程，注入检索 Skill（见 6.4）
  ↓
子 Agent 多轮调用 knowledge_list / knowledge_read
  ↓
子 Agent 综合相关条目，生成结构化答案
  ↓
返回：{ answer: str, sources: list[str] }
```

answer 是综合推理后的答案，sources 是引用的条目文件名列表，调用方可以用 knowledge_read 进一步验证。

#### knowledge_list() → list[str]

返回知识库中所有条目的文件名列表（entries/ 和 placeholders/ 合并，不含 .md 后缀）。等价于：

```bash
ls entries/ placeholders/ | sed 's/\.md$//'
```

供调用方 Agent 自主推理相关文件名，是自主探索路径的入口。

#### knowledge_read(filename: str) → str

返回指定条目的完整 Markdown 内容。filename 不含 .md 后缀，Server 自动在 entries/ 和 placeholders/ 中查找。

### 6.4 OpenCode Skill：知识库探索协议

此 Skill 有两个用途：
1. 分发给用户 Agent，描述如何使用 knowledge_list + knowledge_read 自主探索（路径 B）
2. 作为 knowledge_ask 内部子 Agent 的 system prompt（路径 A 内部复用）

**Skill 内容**：
```
你可以访问一个企业知识库，通过以下工具探索：

工具：
- knowledge_list()：返回所有知识条目名称
- knowledge_read(filename)：读取指定条目的完整内容

探索协议：
1. 调用 knowledge_list() 获取所有条目名称
2. 根据当前问题，推理出 2-5 个语义相关的条目名称
   （文件名是自然语言，使用你的语义理解能力做模糊匹配）
3. 调用 knowledge_read() 逐个读取候选条目
4. 阅读正文中嵌入的 [[链接]]，这些链接在上下文中自然交代了
   "为什么提到它"；也可参考"## 相关"区的快速索引，
   继续对这些概念调用 knowledge_read() 深入探索
5. 若当前关键词无法找到相关内容，换用不同角度的词汇重新推理
6. 综合所有相关条目的内容，形成完整回答，并列出引用的条目名称

注意：
- 文件名即概念，"接口权限控制"和"多角色接口优先澄清权限边界"在语义上相关
- 占位文件（状态：占位）说明该概念存在但尚未完整定义，酌情使用
- 正文中嵌入的链接是深入探索的线索，不要只读第一层
- 关注条目的状态标签："事实"可直接引用，"推断"需留意后续可能修正，
  "存疑"说明来源有矛盾，需综合多方观点
- **避免循环跳转**：记录已读取的条目，不要重复读取同一条目。
  当顺着链接跳转遇到 A→B→A 的循环时，停止深入，换其他路径探索
```

### 6.5 Python 脚本：health_check.py（巡检）

**触发方式**：管理员直接执行 `python health_check.py /knowledge-base/`

**输出**：纯文本健康报告，写入 stdout

**实现逻辑**（完全无状态，每次实时计算）：
```python
def run_health_check(kb_path):
    entries = list_entries(kb_path)           # ls entries/
    placeholders = list_placeholders(kb_path) # ls placeholders/
    all_links = extract_all_links(kb_path)    # grep [[]] 全库
    
    # 统计各占位文件被引用次数
    ref_counts = count_refs(placeholders, all_links)

    orphans = find_orphans(entries, all_links)
    
    # 检测红链（链接目标不存在）
    dangling = find_dangling(all_links, entries + placeholders)
    
    # 输出报告
    print_report(entries, placeholders, ref_counts, orphans, dangling)
```

## 8. 设计决策记录（ADR）

### ADR-001：为什么不使用向量数据库

向量数据库引入了额外的基础设施依赖、同步延迟、以及不透明的黑盒检索过程。

本系统的语义理解工作由 LLM 承担：将 knowledge_list() 返回的文件名列表提供给 LLM，让它推理出相关条目名称，再精确读取文件。这个方案结果完全可解释，零额外基础设施。

向量数据库在知识库规模超过数万条、且需要亚秒级检索时才体现优势。本系统目标规模（数千条）不在此范围内。

### ADR-002：为什么整理不自动触发

自动整理会在知识库质量较低时产生大量低质量的归纳结果，反而污染知识库。人工控制整理节奏，等知识积累到足够密度再执行归纳，质量更可控。整理本身需要人工审核，自动触发只会制造审核队列，不节省时间。

### ADR-003：为什么摄入采用"提取优先"

摄入时不追求最优粒度，而是忠实提取知识并保留推理过程。粒度优化推迟到整理阶段处理，这样摄入成本恒定，不随知识库增长而上升。同时避免了"先打碎"策略导致的信息丢失——只保留结论骨架，丢失推导过程和权衡分析。整理阶段在有足够材料后可以更好地判断粒度和关联。

### ADR-004：为什么整理完全无状态

无状态设计消除了"状态文件与实际文件系统不一致"这类潜在 bug。Python 脚本实时扫描文件系统的成本完全可接受（数千个文件的 grep 在现代硬件上毫秒级完成），换来的是系统健壮性和可维护性的显著提升。

### ADR-005：为什么 UC3 使用 MCP 而非 Skill

Skill 是跟随 Agent 实例分发的本地能力描述，每个接入用户都需要持有这份 Skill。MCP 是服务端能力，接入即可用，符合集中部署的架构方向。此外，knowledge_ask 内部需要访问服务器文件系统，必须在服务端执行，MCP 是天然的暴露方式。

### ADR-006：为什么 knowledge_ask 内部调用 OpenCode CLI

knowledge_ask 需要多轮工具调用（list → read → 跳链接 → 再 read），这是一个 Agent 循环，不是简单的函数。OpenCode CLI 天然支持这个模式，不需要自己实现 Agent 循环逻辑。代价是每次调用有冷启动延迟（可接受），以及额外的 LLM 调用成本。

### ADR-007：为什么探索 Skill 同时用于路径 A 和路径 B

两条检索路径（knowledge_ask 内部和用户自主探索）的核心逻辑完全相同：读文件名列表 → 推理相关条目 → 读文件 → 顺链跳转。将这份逻辑写成一份 Skill，knowledge_ask 的内部子 Agent 和用户侧 Agent 都复用它，保证两条路径行为一致，也降低了维护成本。

### ADR-008：为什么选择 Obsidian 格式

知识库的长期价值取决于能否被持续维护。Obsidian 使用标准 Markdown + `[[双向链接]]` 语法，人类可以不借助任何工具直接阅读和编辑，图谱可视化免费获得。文件即真相，没有隐藏状态。不用 Obsidian CLI，直接操作文件系统即可。

## 9. 摄入 Agent 行为规范

### 9.1 原子化原则

一个命题满足以下任一条件则单独成文：
- 可以脱离原文独立理解
- 被其他命题引用时不需要携带大量上下文
- 包含一个明确的条件和结论

当多个命题因果紧密（A 的结论是 B 的前提，或同一推理链条的多个环节）时，允许合并在一个文件中。不需要拆分得特别细，后续有新知识需要链接时，由整理阶段负责拆分重组。

### 9.1.1 压缩原则

知识库是压缩层，不是镜像。摄入的核心判断是：

- **这是知识还是数据？**
  - 知识 = 可迁移的规律、经验、原则
  - 数据 = 具体的值、实现细节、结构元素

- **这条知识是否已经存在于知识库中？**
  - 如果已有条目表述了相同或近似的规律，不产生新条目

如果输入全文没有可提炼的知识，不产生任何条目。宁可少提，不要多记。

### 9.2 来源类型识别

不同类型的输入，提取策略和过滤目标不同：

| 输入类型 | 提取目标 | 过滤目标 |
|---------|---------|---------|
| **设计文档/复盘报告** | 结论、决策依据、教训、权衡分析 | 具体人名、会议时间、版本号 |
| **代码文件** | 注释/docstring 中的经验教训、代码结构暗示的设计决策 | 函数名、变量名、类名、实现逻辑本身 |
| **API/接口文档** | 接口设计规范、权限约束、错误处理原则 | 单个接口的路径、参数列表、返回值 |
| **配置/部署文档** | 环境搭建规律、依赖关系、运维经验 | 具体配置项的值 |
| **表格/数据报表** | 从数据中呈现的模式 | 表格中的具体数据行 |
| **会议记录** | 决策结论及其理由 | 谁说的、讨论过程的流水账 |

### 9.2.1 负向清单

以下内容**绝对不应**作为知识条目：

- 变量名、函数名、类名、文件名
- 配置项的键名和具体值
- 单个接口的签名（路径、参数、返回值）
- 数据表中的具体数值行
- 人名、时间、会议名
- 没有规律性的事实（如"张三负责订单模块"）
- 纯描述性信息（如"系统使用了 Redis"）

### 9.3 文件命名规范

| 类型 | 命名模式 | 示例 |
|------|----------|------|
| 概念类 | 直接使用概念名 | `幂等性.md` |
| 规律类 | 场景 + 结论的短语 | `多角色接口优先澄清权限边界.md` |
| 规范类 | 领域 + 规范 | `接口设计规范.md` |

命名要求：
- 使用 Agent 在推理时自然会想到的词汇
- 避免缩写、内部代号等不通用的表述
- 不超过 30 个汉字或 50 个英文字符

### 9.4 条目内部结构

根据内容类型选择模板：

**经验教训类模板**：
```markdown
# [文件名]

> **核心**：[用 1-3 句话概括核心结论，不使用"本文介绍"等套话]
> **状态**：[事实 | 推断]

[推理过程：用自由文本叙述得出结论的过程、观察到的证据、关键权衡。
相关概念以 [[链接]] 形式自然嵌入正文中，每处链接都要有上下文说明
"为什么提到它"。反面案例或失败教训也在正文中叙述。]

## 上下文（可选）
- **适用场景**：[该知识在什么场景下适用]
- **前提条件**：[使用此知识需要已知什么]
- **不适用场景**：[该知识在什么情况下不适用]
- **环境假设**：[基于什么技术栈、组织架构、业务阶段的假设]

## 相关
- [[条目名]] — 一句话说明关系（前置依赖/应用场景/对比观点等）

## 来源
[从哪些文档提炼而来，使用 [[链接]] 格式]
```

**概念/规范类模板**：
```markdown
# [概念或规范名称]

> **核心**：[用一句话给出定义或规则]
> **状态**：事实

[补充说明：对该定义或规则的进一步解释、示例、或注意事项。
相关概念以 [[链接]] 形式自然嵌入正文中。]

## 上下文（可选）
- **适用场景**：[...
- **不适用场景**：[...

## 相关
- [[条目名]] — 一句话说明关系

## 来源
[从哪些文档提炼而来，使用 [[链接]] 格式]
```

强制约束：
- 核心结论必须存在，不可为空
- 经验教训类必须有推理过程（仅有结论没有推理过程的条目价值极低）
- 概念/规范类不需要推理过程，但必须有补充说明
- 来源必须存在，不可为空
- 链接必须嵌入正文有意义的上下文中
- "## 相关"区提供轻量级快速索引，每条格式：[[条目名]] — 一句话说明关系

### 9.5 状态标签判断规则

| 状态 | 判断条件 | 示例 |
|------|---------|------|
| **事实** | 原文有明确表述，Agent 直接提炼，无需推断 | "系统在订单超时后自动取消" |
| **推断** | 从多处证据中归纳得出，或 Agent 基于观察推理出原文未明说的规律 | "多个故障复盘都指向同一根因模式，推测这是架构设计缺陷" |

注意：**"存疑"状态不由 ingest 产生**。当 Agent 发现不同文档对同一概念有矛盾解释时，分别按各自文档生成条目（各自标注事实或推断），由 tidy 阶段的"消解矛盾"子动作扫描全库、识别矛盾并创建"争议"区。

### 9.6 链接嵌入规则

将相关概念以 `[[链接]]` 形式嵌入推理过程正文，而非在底部集中列出。

标记为 `[[候选链接]]` 的条件：
- 是专有名词或领域概念
- 在文档中被直接使用但没有内联解释
- 不了解这个词会影响对当前命题的理解

每处链接必须出现在有意义的上下文中：

```
好的写法：
这里的核心是权限边界的定义顺序问题——必须先有[[权限模型]]，
才能讨论接口层面的实现。

不好的写法（底部集中列出）：
## 关联
[[权限模型]] [[接口设计规范]]
```

不标记的情况：
- 通用词汇（"系统"、"用户"、"数据"等）
- 已在当前条目内完整解释的词汇

## 10. 未来扩展方向

### 10.1 知识版本历史

结合 Git 对知识库进行版本管理，每次整理形成一个 commit，支持追溯知识的演化过程。

### 10.2 跨团队知识共享

知识库支持导出为标准格式，允许不同团队之间共享经过验证的知识条目。实现从"个人经验 → 团队财富 → 企业资产"的知识沉淀路径。

### 10.3 知识有效性反馈

Agent 使用知识后，记录该知识是否帮助完成了任务。长期积累后，可以识别出高价值知识和低价值知识，指导整理优先级。

### 10.4 支持更多 Agent 平台

当前仅支持 OpenCode。后续可扩展支持 Claude Code、Codex 等其他平台，MCP Server 已经是标准化接口，新平台接入只需适配 Skill 格式。
